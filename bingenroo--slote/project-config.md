## Context and search (MANDATORY: Context Mesh first)

This project uses **Context Mesh** (MCP server). **You MUST call at least one Context Mesh MCP tool before using Read or Grep.** This is a BLOCKING REQUIREMENT, not a suggestion.

### Efficiency rules

1. **Prefer combo tools** — `get_symbol_context` replaces `get_symbol` + `get_callers` + `get_dependents` in ONE call. Always use it when you need more than just a definition.
2. **Prefer CM file tools** — `list_files` and `read_lines` work from the index (no disk I/O). `list_files` accepts a **substring** of the path/basename when the pattern has no glob characters (`*` `?` `[`); otherwise it uses **glob** matching. Use them instead of Glob/Read when possible.
3. **Call independent tools in parallel** to minimize round-trips.

### Tool selection by scenario

#### Understand code

| Scenario | Tool(s) |
|---|---|
| Understand a file | `get_file_summary(path)` + `get_dependencies(path)` |
| Understand a module/directory | `get_module_summary(module)` |
| Find a symbol definition | `get_symbol(name, include_body=true)` |
| Symbol + usages + dependents (combo) | `get_symbol_context(name)` — ONE call instead of three |
| Find usages/call sites only | `get_callers(name)` |
| What depends on this file? | `get_dependents(path)` |
| Find tests for a file | `get_tests_for(path)` |

#### Trace and analyze

| Scenario | Tool(s) |
|---|---|
| Trace a user-facing flow (natural language) | `explain_flow("user login flow")` |
| Call chain between two functions | `trace_call_chain(source, target)` |
| BFS walk around a symbol | `expand_context(symbol)` |
| Impact analysis (what breaks?) | `impact_analysis_deep(symbol)` — transitive caller walk |
| Compare 2+ file structures | `compare_files(files, mode="structure")` |
| Diff 2+ function implementations | `compare_symbols(symbols, mode="diff")` |

#### Discover and search

| Scenario | Tool(s) |
|---|---|
| Open-ended / broad question | `search_code("question")` |
| List files by pattern | `list_files(pattern)` — index-backed; substring or glob (see efficiency rules) |
| Read specific lines | `read_lines(path, start, end)` — partial-path match, no Read needed |
| Codebase stats (size, languages) | `get_stats()` |
| Frameworks, deps, build tools | `get_ecosystem_context()` |
| API endpoints | `get_routes()` |

#### Memory (cross-session persistence)

| Scenario | Tool(s) |
|---|---|
| Recall past decisions/context | `recall(query)` |

### Fallback chain

If your first Context Mesh call returns no results (e.g. `get_symbol` → "not found"):
1. Try `search_code` with a descriptive query — do NOT immediately fall back to Grep/Read.
2. Only if `search_code` also fails, then Grep/Read is permitted.

### After Context Mesh

- Use `Read` only for the specific line ranges identified. Use offset/limit — never read whole files unless short (<50 lines).
- After `get_file_summary`, prefer `get_symbol_context`/`get_callers` to navigate to related code. Only use Read for the final specific lines that Context Mesh cannot provide.
- Grep is allowed for narrow, targeted searches on files/lines already identified by Context Mesh.

### When Read/Grep without Context Mesh is OK
- You are re-reading lines you already explored with Context Mesh in this conversation.
- You are reading a file the user explicitly pointed you to (e.g. opened in IDE, pasted a path).

## Edit behavior
- Before making any edit, always show the before/after code snippets and link to the specific lines being changed (e.g. [file.cpp:42](src/file.cpp#L42)).
- In read-only or Ask mode, supply patches instead of applying edits; when edit tools are available (e.g. Agent mode), apply changes directly.

## Parallel sub-agents

When a task touches 3+ independent areas (e.g., separate modules, unrelated directories):
- Spawn parallel Agent sub-agents, one per area, each using Context Mesh tools independently.
- Combine results in the parent after all complete.
- Do NOT parallelize sequential/dependent changes.

## Task tracking

For multi-step tasks (5+ files or 3+ distinct changes), use TodoWrite to:
- Break work into tracked steps at the start.
- Mark each step complete as you go.
- This prevents losing track of progress in long conversations.
