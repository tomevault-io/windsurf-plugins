"Lessons learned:

- **Constructor Dependency Inflation**: Adding a new dependency to a core service (like `TranslationRequestService`) will break all unit tests that manually instantiate that service. Always grep for usages of `new [ServiceName]` in the `*.Tests` project after modifying a constructor to catch and fix broken tests early.
- **Nullability and XML Warnings**: CI environments often treat warnings as errors or run stricter analysis. Proactively fixing `CS8604` (possible null reference) and XML comment mismatches in the files you modify prevented downstream build failures in this task.
- **Vue Type Checking**: Standard `npm run build` behavior can be inconsistent across environments. Use `npx vue-tsc --noEmit` for a reliable, standalone type-check of the frontend during validation.
- **Repair Phase Resilience**: The translation repair phase is naturally prone to handling 'hard-to-translate' lines that may cause LLM output issues like JSON truncation. Repair logic must always implement the same chunking and fallback splitting protection as the main translation loop to avoid cascading job failures when the repair batch itself exceeds provider limits.
- **Vue Infinite Reactivity Loops**: When syncing two reactive objects (e.g., `visibleLayout` and `currentLayout` in a drag-and-drop grid), using deep watchers that modify each other's source data will cause an infinite rendering loop, freezing the browser tab instantly. Always use strictly decoupled comparison (like `JSON.stringify`) or emit/event-based one-way data flow to break the dependency cycle. Also, when saving layout arrays, ensure hidden or omitted items are securely merged rather than completely overwritten to prevent data destruction.
- **Cleanup Operations Must Close the Loop**: When a cleanup/migration service modifies data (e.g., reassigning `SourceInstanceId` to `"default"`), it MUST also update the configuration that feeds the data pipeline (e.g., instance settings). Otherwise the next pipeline run (sync job) will re-tag data with the old value, creating an infinite cleanup cycle. Always trace the full data lifecycle: `settings → job → DB → cleanup → settings` and verify the loop closes.
- **Conditional Branch Coverage for Edge Cases**: Guard conditions like `Count > 1` silently exclude the most common user scenario (single instance). Before committing any fix, manually trace through each real-world scenario (single instance, multiple instances, zero instances) against every conditional branch to verify correctness. The `Count > 1` bug would have caused the cleanup to silently fail for most users.
- **Vue 3 v-if/v-else-if Chain Breaking**: HTML comments (`<!-- Widget Name -->`) between `<template v-if>` and `<template v-else-if>` blocks **break the Vue 3 directive chain**. The Vue compiler treats comments as sibling elements, disconnecting `v-else-if` from its preceding `v-if`. Use independent `v-if` checks directly on components instead of `<template>` wrappers with `v-else-if` when comments are present between them. This caused 4 dashboard widgets to silently render as empty comment nodes.
- **Unique Constraint Violations During Bulk Reassignment**: When using `ExecuteUpdateAsync` to bulk-reassign a foreign key (e.g., `SourceInstanceId`), always check for unique constraint collisions FIRST. If the target value already exists for the same natural key, the bulk update will fail with a constraint violation. Delete colliding records first, then reassign the remainder.
- **Vue Named Slot vs Default Slot Mismatch**: When a parent component (`CardComponent`) only defines named slots (`#content`, `#icon`, `#description`) and no default `<slot />`, child content placed without a `<template #slotName>` wrapper is silently discarded. Always check the parent component's slot definitions when new child components render their frame/title but show empty content areas. This caused all 4 dashboard widgets to show titles but no data.
- **TypeScript Type Safety for i18n**: When expanding language support, update the `ILocale` type definition FIRST to include all supported languages. The type system prevents runtime errors but outdated types (like having only `en` and `nl` when 7 languages exist) create false confidence. Always verify that `LOCALE` enum and `ILocale` type match the actual languages returned by the backend `/api/translation/languages` endpoint.
- **Browser Language Detection Priority Chain**: Implement a proper fallback chain for locale detection: localStorage (user's explicit choice) → backend setting (persistent preference) → browser language (first visit auto-detection) → default fallback ('en'). This respects user choice while providing a seamless first-time experience for international users. Never rely solely on browser detection as users on shared computers may have different preferences than the browser's language setting.

- **Grid-Layout-Plus Responsive Saving**: When using `<GridLayout v-model:layout="currentLayout" :responsive="true">`, the library will automatically scale down widget dimensions (`w`, `h`) inside `currentLayout` when the window shrinks to smaller breakpoints. If you have a deep watcher syncing `currentLayout` back to the database, this will permanently overwrite your base layout with the squashed mobile layout! When the window expands back, it uses the squashed layout as the new baseline, causing widgets to get "tighter and tighter" every time you resize. **Fix:** Only save layout changes when the user is explicitly in an "Edit/Config" mode (`if (!isConfigMode) return;` in the watcher). Never attempt to fix this by destroying/recreating the grid (`key += 1` with a `ResizeObserver`), as this causes calculation failures during CSS animations.

- **TranslationState Statistics and Re-evaluation**: When relying on `TranslationState.Complete` for frontend dashboard statistics (like progress bars), NEVER bulk-update these items to `TranslationState.Stale` during a language settings change. Doing so will immediately drop the user's dashboard progress to 0% until the background job re-evaluates all items, which looks like a bug to the user. Instead, keep them as `Complete` and use a `StateSettingsVersion` integer on the entity. Increment the version globally on settings change, and have the background job query for `StateSettingsVersion < CurrentVersion` to gracefully re-evaluate items in the background without causing transient UI failures.
