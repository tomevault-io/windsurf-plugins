## Writing Style

### Tone and Voice
- **Conversational and direct** - Use first person ("I") and second person ("you") frequently
- **Pragmatic over theoretical** - Focus on practical solutions, not academic discussions
- **Honest about limitations** - Always mention trade-offs, gotchas, and when something "is not perfect"
- **Humble and helpful** - Phrases like "If you need help, feel free to reach out"
- **No-nonsense** - Get to the point quickly, avoid fluff

### Structure Patterns

**Opening:**
- Start with a brief problem statement or context (1-2 paragraphs)
- Often include: "Straight to the code" with a GitHub link at the very beginning
- Use rhetorical questions or scenarios: "So you have a multistage docker build..."

**Body:**
- Use clear H3 (###) section headings
- Break down complex topics into digestible chunks
- Each section should be actionable and self-contained
- Include step-by-step instructions with code examples
- Use bullet points for lists of benefits, requirements, or steps

**Code Examples:**
- ALWAYS include working code, not pseudocode
- Show complete examples, not fragments
- Link to full GitHub repositories with working examples
- Use code blocks liberally with proper syntax highlighting
- Include comments in code to explain non-obvious parts
- Show "before and after" comparisons when optimizing

**Visual Aids:**
- Use tables for comparisons (especially pricing, specs)
- Include command-line output examples
- Reference images/diagrams when they add value

**Closing:**
- End with "Wrapping Up", "Last words", or "Final Considerations"
- Summarize key takeaways
- Encourage readers to try the solution
- Offer to help: "feel free to reach out"

### Technical Writing Guidelines

1. **Keep it simple** - Follow YAGNI principle
2. **Be specific** - Use exact version numbers, specific tool names, real examples
3. **Link generously** - Link to official docs, GitHub repos, related articles
4. **Acknowledge complexity** - If something is tricky, say so
5. **Provide context** - Explain WHY before diving into HOW
6. **Show, don't just tell** - Always pair explanations with code/examples

### Common Patterns

- **Build tags and conventions**: Explain Go-specific patterns like `//go:build servicetest`
- **Configuration**: Show environment variable setup, config files
- **Testing**: Include test commands with flags explained
- **Docker**: Show both Dockerfile and build commands
- **Real-world examples**: Use actual open-source projects (go-ethereum, etc.)

### Language and Phrasing

**Preferred phrases:**
- "Here is what we need to do:"
- "Let's test it"
- "Things to keep in mind:"
- "As you can see..."
- "The idea is simple."
- "That's it."
- "Not bad, huh?"

**Avoid:**
- Marketing speak or hype
- Overuse of emojis (use sparingly, like ⚠️ for warnings)
- Long introductions
- Academic language
- Unnecessarily complex explanations

### Article Length
- Aim for 1500-3000 words (medium-length articles)
- Focus on one main topic with 5-8 subsections
- Include at least 3-5 code examples
- Link to a complete working GitHub repository

## Content Guidelines

### When Writing New Articles

1. **Start with a problem** - What pain point are we solving?
2. **Link to working code** - Create or reference a GitHub repo with full example
3. **Show the solution incrementally** - Build up from simple to complete
4. **Include all the pieces** - Dockerfile, CI config, deployment specs, etc.
5. **Test the code** - All examples should actually work
6. **Add practical context** - Mention real-world usage, pricing, limitations
7. **Keep it focused** - One main topic per article

### Topics to Avoid
- Front-end development (not the focus area)
- Purely theoretical computer science
- Beginner tutorials (assume reader knows basics)
- Product reviews without technical depth
- Generic "how to code" content

## Special Patterns

### Disclaimers and Notes
Use disclaimers when needed, formatted as:
- *Italicized notes* for clarifications
- ⚠️ emoji for warnings about limitations
- "Things to keep in mind:" for important caveats

### Repository References
- Create example repos under DenisPalnitsky or num30 GitHub accounts
- Use descriptive repo names: `github-actions-advanced-techniques`, `golang-service-test`
- Include README with clear instructions

### Code-First Approach
- Show the code first, explain after
- Full working examples over snippets
- Link to specific files in repos using GitHub URLs

### Measurements and Benchmarks
- Include before/after comparisons
- Show actual numbers: "10 times faster", "$0.008/minute"
- Use real-world examples with tangible results
