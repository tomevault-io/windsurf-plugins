# AGENTS.md

## 1. 项目定位

这是一个 **产品原型仓库**，不是可交付的业务系统。

目标：
- 用于快速生成和迭代 HiGood 相关系统页面原型
- 风格接近 Axure 输出：重 UI、重信息结构、重交互演示、重 Mock 数据
- 通过 GitHub 自动部署到 Vercel 供在线查看

默认原则：
- 只做页面结构、轻交互、Mock 数据
- 不做复杂工程化架构
- 不做真实后端、真实权限、真实状态机、真实数据库设计
- 不做 API 对接、状态管理体系重构、领域模型重构
- 不做“为未来可扩展”而提前引入的复杂抽象

---

## 2. 技术栈

默认技术栈固定为：

- Vite
- TypeScript
- Tailwind CSS

默认不引入：
- Next.js
- Redux / Zustand / MobX / Provider 架构
- React Query / SWR / 复杂远程数据层
- 后端服务、接口层、鉴权体系
- 复杂表单引擎
- 复杂图表库（除非页面明确需要）

说明：
- 这是原型项目，不是工程能力展示项目
- 优先静态页面 + 本地 Mock 数据 + 轻量交互
- 所有页面都应以“看起来像真实系统”而不是“实现真实系统”为目标

---

## 3. 全局工作方式

Codex 默认按 **增量 patch** 工作，不要大范围重写。

每次任务默认遵循：
1. 先理解当前页面/模块现状
2. 只修改与当前任务直接相关的文件
3. 优先复用已有样式、布局、组件
4. 不主动重构无关代码
5. 不主动优化与当前任务无关的目录结构
6. 不主动升级依赖
7. 不主动引入新的基础设施

除非用户明确要求，否则：
- 不新建大型通用组件体系
- 不抽象“未来可能复用”的工具层
- 不把页面原型改造成工程化项目

---

## 4. 输出与改动边界

每次改动必须遵守：

- 优先小步修改
- 优先局部替换
- 优先最少文件改动
- 优先保留现有可运行页面
- 不做整页重写，除非用户明确要求
- 不修改无关模块
- 不把一个小需求扩写成系统级重构

如果需求较大，默认拆成多轮 patch：
- 先做结构
- 再补交互
- 再补 Mock 数据
- 最后再做收口优化

---

## 5. 原型设计原则

### 5.1 页面目标
所有页面必须优先表达：
- 信息结构
- 业务关系
- 状态变化
- 页面跳转
- 关键动作入口
- 演示所需的 Mock 数据

### 5.2 不做的东西
默认不做：
- 真实业务校验已完成
- 复杂接口联调
- 复杂状态同步
- 复杂缓存
- 复杂错误恢复机制
- 复杂上传系统
- 复杂图像处理
- 复杂权限控制

### 5.3 Mock 数据原则
Mock 数据必须：
- 足够支撑页面阅读和交互
- 能体现上一步与下一步关系
- 能体现不同状态
- 能体现异常/风险/边界场景
- 不要只有空壳页面
- 不要只有一两条极少数据

Mock 数据应优先表现：
- 中文业务语义
- 真实字段组合
- 状态差异
- 典型异常

---

## 6. 中文化要求

这是中文业务原型项目，必须严格遵守：

- 页面文案全部中文展示
- 状态值全部中文展示
- 禁止英文状态码直接出现在页面上
- 禁止出现乱码
- 时间、金额、单位、状态统一用中文业务语义表达

例如：
- 不显示 `PENDING`, `DONE`, `IN_PROGRESS`
- 要显示 `待处理`、`已完成`、`进行中`

---

## 7. UI 与交互要求

默认遵循以下原则：

- 风格简洁、清晰、偏企业后台
- 优先可读性，不追求炫技
- 页面布局稳定，不频繁变换交互范式
- 保持模块内风格统一
- 轻交互优先：抽屉、弹窗、页签、筛选、详情展开
- 不做复杂动画
- 不做重视觉营销风格

默认要关注：
- 列表是否可读
- 卡片是否能承载关键信息
- 筛选是否足够用
- 详情是否能承接列表入口
- 页面之间是否能连起来

---

## 8. 目录与模块改动规则

默认只改与当前需求相关的页面和 mock 数据文件。

如果需求没有明确要求：
- 不改全局 layout
- 不改路由总结构
- 不改无关菜单
- 不改无关模块
- 不改项目脚手架配置
- 不改部署配置

只有在以下情况下才允许改全局：
- 当前需求明确要求
- 不改就无法完成页面接入
- 改动范围可控且必要

---

## 9. 模块实现约定

### 9.1 FCS / HiGood 页面
这些页面默认是产品原型，不是生产系统：
- 侧重页面表达
- 侧重链路展示
- 侧重信息和状态
- 侧重跨页面跳转
- 侧重演示价值

### 9.2 字典类页面
字典页默认按 **结果展示页** 处理，不做重后台维护感。
除非用户明确要求，否则：
- 不默认开放新增
- 不默认开放批量操作
- 不默认开放复杂编辑能力
- 优先展示结果、来源、引用关系

### 9.3 技术包/任务/PDA 页面
默认：
- 重点是页面可读性和业务已完成演示
- 用 mock 数据体现来源、状态、上一步与下一步
- 不引入复杂底层逻辑

---

## 10. 任务执行默认格式

当用户让 Codex 改页面时，默认按以下方式执行：

1. 先识别本次只涉及哪些页面/文件
2. 只在相关范围内 patch
3. 补足必要 Mock 数据
4. 保持页面可演示
5. 不扩大到无关模块
6. 最终给出：
   - 改了哪些文件
   - 做了什么
   - 如何验证

---

## 11. 验证原则

每次改动后，尽量验证：
- 页面是否能正常运行
- 路由是否可达
- 交互是否可点
- 列表与详情是否连通
- 不出现明显空白页、死链、报错页
- 中文文案和状态是否正确

如果项目已有可用命令，则优先运行最小必要验证。

如仓库命令已存在，请使用：
- 开发启动：`npm run dev`
- 构建检查：`npm run build`

如果实际命令不同，请以仓库现有脚本为准，不主动新造命令。

---

## 12. 明确禁止事项

除非用户明确要求，否则禁止：

- 把原型项目改造成工程化架构样板
- 大规模重构目录
- 引入 store / provider / API 层
- 引入真实服务端依赖
- 为“以后可能用到”提前做复杂封装
- 用英文状态码直接出页面
- 用大段无意义静态假数据灌满页面
- 把一个小 patch 做成系统级改造

---

## 13. 与用户协作的默认原则

默认认为用户要的是：
- 快速、准确、低成本地迭代原型
- 可持续 patch，而不是推翻重来
- 结果可直接部署到 Vercel 演示
- 页面与业务一致，而不是代码“看起来高级”

因此：
- 优先满足原型准确性
- 优先满足业务表达
- 优先控制改动范围
- 优先控制复杂度
