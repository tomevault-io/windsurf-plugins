# Gemini 開発ルール

このドキュメントは、すべてのプロジェクトに適用されるグローバルな開発ルールを概説します。
Gemini CLIは、普遍的な開発標準の主要な情報源としてこのファイルを使用します。

## 1. コミットメッセージ

すべてのコミットメッセージは、Conventional Commits仕様に準拠する必要があります。
これにより、バージョニングと変更履歴の自動生成が保証されます。

- **フォーマット**: `<type>: <subject>`
- **例**: `feat: ユーザー認証エンドポイントの追加`
- **type一覧**: `commit-types.yaml` を参照してください

## 2. セキュリティ

- **コードにシークレットを含めない**: APIキー、パスワード、その他のシークレットをコードベースに直接コミットしないでください。環境変数またはシークレット管理システムを使用してください。
- **依存関係の監査**: サードパーティの依存関係を定期的に監査し、脆弱性を特定して修正してください。

## 3. ブランチモデル

すべてのリポジトリでGitFlowブランチモデルに従ってください。
機能開発は必ず、developブランチから切るようにしてください。

### ブランチの種類
- `main`: 本番環境向けのコード。
- `develop`: 次期リリースのための統合ブランチ。
- `feature/*`: 新機能開発用。
- `bugfix/*`: バグ修正用。
- `hotfix/*`: 緊急の本番環境の修正用。

### 新機能開発の手順
1. **developブランチに切り替え**: `git checkout develop`
2. **最新の状態に更新**: `git pull origin develop`
3. **新機能ブランチを作成**: `/new-feature <機能名>` コマンドを使用
   - 例: `/new-feature user-authentication`
   - このコマンドは自動的に `feature/<機能名>` ブランチを作成し、基本的なファイル構造をセットアップします
4. **開発作業を開始**: 作成されたブランチで機能開発を進める

**注意**: 手動でブランチを作成する場合は、必ず `feature/` プレフィックスを使用し、developブランチから作成してください。

## 4. コード品質

### リンターエラーの修正

**すべてのリンターエラーは、コミット前に必ず修正してください。**

- **自動修正可能なエラー**: `npm run lint:fix` または `yarn lint:fix` を実行して自動修正
- **手動修正が必要なエラー**: エラーメッセージを確認し、適切な修正を実装
- **エラーが0件になるまで**: リンターエラーが完全に解消されるまでコミットを行わない
- **CI/CDでのチェック**: プルリクエスト作成時にもリンターエラーがないことを確認（CI/CDパイプラインで自動実行）

**修正手順:**
1. **Gemini CLIでリンターチェックを実行**: `/lint-check` コマンドを実行
   - プロジェクトの言語に応じたリンターチェックと自動修正が実行されます
2. **手動修正が必要なエラーを個別に対応**: 自動修正できないエラーがあれば手動で修正
3. **再度リンターチェックを実行**: `/lint-check` コマンドを再実行してエラーが0件であることを確認
4. **エラーが解消されたらコミットを実行**: すべてのリンターエラーが解消されたことを確認してからコミット

**手動での実行（Gemini CLIが使用できない場合）:**
- **自動修正可能なエラー**: `npm run lint:fix` または `yarn lint:fix` を実行して自動修正
- **手動修正が必要なエラー**: エラーメッセージを確認し、適切な修正を実装
- **エラーが0件になるまで**: リンターエラーが完全に解消されるまでコミットを行わない
