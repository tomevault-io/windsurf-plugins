## Project Overview

This is a PyTorch Lightning + Hydra based deep learning project for NoXiRe (Rapport Recognition). The codebase follows the Lightning-Hydra-Template structure for organized ML experimentation with config-driven development.

## Architecture

### Config-Driven Design (Hydra)

All configuration is managed through Hydra configs in `configs/`. The system uses **dynamic instantiation** - Python objects are created from config files at runtime using `_target_` paths.

**Key concept**: Instead of importing and instantiating classes directly in code, you specify the module path in YAML:

```yaml
# configs/model/mnist.yaml
_target_: src.models.mnist_module.MNISTLitModule
lr: 0.001
net:
  _target_: src.models.components.simple_dense_net.SimpleDenseNet
  input_size: 784
  output_size: 10
```

Then instantiate with: `model = hydra.utils.instantiate(config.model)`

### Directory Structure

```
configs/               # All Hydra configuration files
├── callbacks/        # Lightning callbacks (checkpointing, early stopping, etc.)
├── data/            # DataModule configs
├── model/           # Model configs
├── trainer/         # Lightning Trainer configs (GPU/CPU, DDP, etc.)
├── logger/          # Logger configs (W&B, TensorBoard, etc.)
├── experiment/      # Experiment-specific overrides
├── debug/          # Debug mode configs
└── train.yaml      # Main training config (composes above configs)

src/
├── data/           # LightningDataModule implementations
│   ├── components/ # Dataset classes
│   └── *_datamodule.py
├── models/         # LightningModule implementations
│   ├── components/ # Neural network architectures
│   └── *_module.py
└── utils/          # Utility functions (logging, instantiation, etc.)

data_preparation/   # Data preprocessing scripts (separate from training)
├── hdf5_pipeline/  # HDF5 conversion for efficient data loading
└── skeleton_testing/ # Skeleton data visualization/testing

tests/             # pytest tests
logs/              # Training outputs (generated by Hydra)
```

### Training Pipeline Flow

1. **Config Composition** (`src/train.py`): Hydra composes config from multiple YAML files
2. **Instantiation**: DataModule, Model, Callbacks, Logger, Trainer created from configs
3. **Training**: `trainer.fit(model, datamodule)` using Lightning's training loop
4. **Testing** (optional): `trainer.test()` on test set with best checkpoint
5. **Logging**: All artifacts saved to `logs/<task_name>/runs/<timestamp>/`

### Lightning Components

**LightningDataModule** (e.g., `src/data/mnist_datamodule.py`):
- Handles all data operations: download, prepare, setup, dataloaders
- Methods: `prepare_data()`, `setup()`, `train_dataloader()`, `val_dataloader()`, `test_dataloader()`

**LightningModule** (e.g., `src/models/mnist_module.py`):
- Defines model, loss, optimization, and training/val/test steps
- Methods: `__init__()`, `forward()`, `training_step()`, `validation_step()`, `configure_optimizers()`
- Use `self.log()` to log metrics (automatically handled by Lightning)

**Trainer**: Manages training loop, GPU/TPU, distributed training, callbacks, logging

### Key Utilities

**`rootutils.setup_root()`** (in `src/train.py`):
- Adds project root to PYTHONPATH (enables `from src import ...`)
- Sets PROJECT_ROOT environment variable
- Loads `.env` file
- Uses `.project-root` marker file to find project root

**`src/utils/instantiators.py`**:
- `instantiate_callbacks()`: Creates Lightning callbacks from config
- `instantiate_loggers()`: Creates loggers from config

**`src/utils/utils.py`**:
- `task_wrapper`: Decorator for error handling in multiruns
- `log_hyperparameters()`: Logs all config to loggers
- `get_metric_value()`: Safely retrieves optimization metric

## Data Preparation Pipeline (NoXiRe-Specific)

This project includes a comprehensive data preparation system in `data_preparation/`:

### Skeleton Data Processing
- **Input**: Kinect binary `.skel.stream~` files (350 dimensions: 25 joints × 14 features)
- **Processing**: `format_conversion.py` converts to numpy arrays
- **Output**: `.npy` files with shape (T, 350) or (T, 25, 14)
- **Testing**: `skeleton_testing/test_stream_conversion.py`

### HDF5 Pipeline (`data_preparation/hdf5_pipeline/`)
Efficient storage format for large multimodal datasets:

**Key files**:
- `data_hdf5_flexible.py`: Main conversion script (supports OpenFace, ViT, or mixed features)
- `extract_vit_features.py`: Extract vision transformer embeddings
- `pytorch_example.py`: Reference Dataset/DataLoader implementations

**Why HDF5**: 50-100× faster random access, 40-60% smaller size, direct frame slicing

**Normalization strategy**:
- **OpenFace features**: Min-max scaling to [0, 1] (handcrafted features benefit from normalization)
- **ViT features**: No normalization (already normalized by LayerNorm, scaling would distort learned embeddings)
- See `hdf5_pipeline/NORMALIZATION_GUIDE.md` for detailed explanation

## Config System Patterns

### Experiment Configs
Store versioned hyperparameters in `configs/experiment/`:
```yaml
# @package _global_

defaults:
  - override /data: my_data
  - override /model: my_model

tags: ["experiment_name"]
seed: 12345

trainer:
  max_epochs: 100

model:
  lr: 0.001
```

Run with: `python src/train.py experiment=my_experiment`

### Local Machine Configs
Use `configs/local/default.yaml` for machine-specific paths (excluded from git):
```yaml
# @package _global_

paths:
  data_dir: /your/local/path/data/
```

### Environment Variables
Store private keys/paths in `.env` (excluded from git):
```
WANDB_API_KEY=your_key_here
DATA_ROOT=/path/to/data
```

Reference in configs: `path: ${oc.env:DATA_ROOT}`

## Common Patterns

### Adding a New Model
1. Create model class in `src/models/components/my_model.py`
2. Create Lightning module in `src/models/my_module.py`
3. Create config in `configs/model/my_model.yaml` with `_target_: src.models.my_module.MyModule`
4. Run: `python src/train.py model=my_model`

### Adding a New Dataset
1. Create dataset class in `src/data/components/my_dataset.py`
2. Create datamodule in `src/data/my_datamodule.py`
3. Create config in `configs/data/my_data.yaml` with `_target_: src.data.my_datamodule.MyDataModule`
4. Run: `python src/train.py data=my_data`

### Accessing DataModule Attributes in Model
**Option 1**: Pass as init parameter
```python
# src/train.py
datamodule = hydra.utils.instantiate(config.data)
model = hydra.utils.instantiate(config.model, num_classes=datamodule.num_classes)
```

**Option 2**: Variable interpolation in config
```yaml
# configs/model/my_model.yaml
_target_: src.models.my_module.MyModule
num_classes: ${data.num_classes}
```

**Option 3**: Access via trainer in Lightning hooks
```python
# In LightningModule
def on_train_start(self):
    self.num_classes = self.trainer.datamodule.num_classes
```

### Logging Metrics
```python
# In LightningModule
def training_step(self, batch, batch_idx):
    loss = self.criterion(preds, targets)
    acc = self.train_acc(preds, targets)

    # Log to all loggers
    self.log("train/loss", loss, on_step=True, on_epoch=True)
    self.log("train/acc", acc, on_step=False, on_epoch=True)

    return loss
```

Use `/` in metric names to organize in logger UI (e.g., `train/loss`, `val/loss`)
