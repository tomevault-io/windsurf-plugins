# TSP Development Workflow

You are working in a TSP (The Software Practice) project. Follow this spec-driven workflow for all work.

## Context Gathering Protocol

Before starting any non-trivial task, read project context in this order:

1. `README.md` — project purpose, architecture, setup
2. `architecture.md` — architectural boundaries, key flows, invariants (if present)
3. `features.md` — what's already built (grouped by module)
4. `impldocs/INDEX.md` — implementation document registry with status and summaries
5. `roadmap.md` — planned work and priorities
6. `todos.md` — deferred work and known issues
7. `.tsp-copilot/cache/code-index.md` — file inventory with descriptions (if present; ephemeral, generated by hooks)

`architecture.md` is human-authored and AI-read-only. **Never modify it.** If it is missing, proceed without it. If its Change Log contains only the template entry (`Empty — initial scaffold`), treat it as empty and mention to the developer that filling it in would improve planning and review quality.

**Do NOT read individual impldoc files unless INDEX.md indicates they are relevant to the current task.** This keeps context lean. When you need detail on a specific feature, read only that impldoc.

Check `todos.md` for items that can be addressed as part of the current work. Suggest these to the developer but do not act on them without approval.

## Spec-Driven Development

Every feature or significant change requires an implementation document (impldoc) before any code is written. Bug fixes and small changes do not require impldocs — use standard workflow with conventional commits.

### Decomposition Principle

Features should target the **smallest useful independently verifiable slice**. Each slice must be understandable, implementable in one cycle, testable, and reviewable. If a feature is too broad to fit comfortably in one impldoc, split it into sequential impldocs — each delivering a working increment. The planner will actively recommend splitting when scope signals suggest it; the developer makes the final call.

### Starting a New Feature

All new features **must** go through the `tsp-planner` agent. Do not start implementation directly.

1. The developer opens `tsp-planner` (or uses the `/tsp-new-feature` prompt)
2. `tsp-planner` reads project context per the protocol above
3. `tsp-planner` interviews the developer — asking about requirements, constraints, edge cases, and affected areas before drafting anything
4. Once the conversation reaches sufficient depth, `tsp-planner` drafts `impldocs/{name}-{suffix}.md` (e.g., `user-authentication-a1.md`)
5. The developer reviews and approves the impldoc
6. Only then does the developer switch to agent mode (or `tsp-implementer`) for implementation

### Implementing a Feature

Use the `tsp-implementer` agent (or the `/tsp-implement` prompt) to implement an approved impldoc. The implementer orchestrates the full workflow:

1. Reads the impldoc and implements code step-by-step
2. Delegates to `tsp-linter` for formatting and warning fixes
3. Delegates to `tsp-unit-tester` for test creation and coverage
4. Delegates to `tsp-code-reviewer` and `tsp-red-team` for quality and security review
5. Delegates to `tsp-doc-updater` for documentation updates
6. Reports completion with a summary

The implementer follows these rules:
- Follow the approved impldoc precisely — implement what it specifies
- Do not add unrequested features, refactors, or "improvements"
- Maximum 3 fix cycles per quality gate — surface unresolved issues to the developer
- If the impldoc needs changes during implementation, follow the Refinement Protocol below

### Refinement Protocol

During implementation, requirements often evolve through back-and-forth conversation. When changes arise:

1. **Update the impldoc first** — before changing code, update the relevant section in the impldoc to reflect the new direction
2. **Log significant deviations** — add an entry to the impldoc's `## Change Log` section for any change that alters requirements, design decisions, or scope
3. **Keep docs in sync** — after a round of refinements, ensure `features.md`, `INDEX.md`, and other project docs still reflect the current state
4. **The impldoc at completion should reflect what was actually built**, not just the original plan
5. **Respect scope boundaries** — if scope changes would expand beyond the original Scope Boundaries section, create a follow-up impldoc rather than inflating the current one. Log the split decision in the Change Log.

## Merge-Readiness Review

After implementation is complete and the implementer has reported its results, use the `tsp-reviewer` agent (or the `/tsp-review` prompt) for a strict, independent merge-readiness review. The reviewer:

- Challenges assumptions and demands justification for non-trivial choices
- Checks impldoc fidelity — did the implementation actually satisfy the spec?
- Identifies evidence gaps — missing tests, missing failure-path validation, missing design rationale
- Produces a review artifact (`impldocs/{id}-review.md`) with a verdict, findings, and guidance for human MR reviewers
