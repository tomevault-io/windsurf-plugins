# LLM Wiki

This is a personal knowledge base system. An LLM (you) reads raw data (writing, tweets, messages, bookmarks) and compiles it into interconnected wiki articles.

## Project Structure

- `.claude/skills/wiki/SKILL.md` — The wiki skill with all commands (`/wiki ingest`, `/wiki absorb`, `/wiki query`, etc.)
- `data/` — Raw source files (user drops data here)
- `raw/entries/` — Individual markdown entries (generated by ingest scripts)
- `wiki/` — Compiled wiki articles (generated by you via `/wiki absorb`)
- `outputs/` — Query results and reports
- `pages/wiki/` — Next.js web viewer
- `components/wiki/` — React components for rendering
- `ingest.py` — Multi-format data ingest (writing, tweets, bookmarks)
- `ingest_imessage.py` — macOS iMessage extractor (reads SQLite directly)
- `ingest_whatsapp.py` — macOS WhatsApp extractor (reads SQLite directly)

## Key Conventions

- Wiki articles use `[[wikilinks]]` for cross-references. Format: `[[directory/slug]]` or `[[directory/slug|Display Text]]`
- Articles have YAML frontmatter: title, type, created, last_updated, related, sources
- Categories (directories in `wiki/`) emerge from the data — don't pre-create them
- The `_index.md` and `_backlinks.json` files are auto-generated — rebuild with `/wiki rebuild-index`

## Absorb Strategy

When absorbing large datasets, process in this order:

1. **Writing first** — Published writing is the highest signal. It establishes the wiki's backbone with well-developed ideas.
2. **Tweets second** — Cluster by theme. Use to extend and reinforce ideas from writing. Only create standalone articles for ideas tweeted about but never written up.
3. **Bookmarks third** — Interest signals only. Don't create articles about bookmarked tweets. Instead, note what topics attract attention and update existing articles.
4. **Messages last** — Highly selective. Only absorb patterns that repeat or moments that clearly mattered. Casual conversations are noise.

For large batches (1000+ entries), use parallel subagents to process different time periods or source types simultaneously.

## Common Gotchas

- `gray-matter` parses YAML dates as `Date` objects. The wiki-loader converts them to strings with `toDateString()` for JSON serialization.
- The `[[wikilink]]` transform runs before the remark pipeline (in `transformWikilinks`). It supports both `[[slug]]` and `[[slug|Label]]` formats.
- Global CSS from `_app.tsx` can interfere with wiki styles. The `.wiki-page` class overrides are in `global.css` — they suppress the `::before`/`::after` pseudo-elements on links and reset heading styles.
- In dev mode, CSP may block React hot reloading. The `next.config.js` adds `'unsafe-eval'` to script-src only in development.
- The `body > div` max-width constraint (if present in global CSS) needs to be overridden for wiki pages using `body:has(.wiki-page) > div { max-width: 100%; }`.

## Writing Standards

When writing wiki articles:
- Wikipedia tone. Flat, factual, encyclopedic.
- Never use: em dashes, "genuine", "deeply", "truly", "legendary", "powerful", "importantly", "profound"
- Attribution over assertion: "The subject argued X" not "X is true"
- Max 2 direct quotes per article
- Organize by theme, not chronology
