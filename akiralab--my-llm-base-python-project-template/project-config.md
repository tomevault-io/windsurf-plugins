# CLAUDE.md

このリポジトリでは、AIエージェント（Claude / Codex）を補助的に利用します。  
最終意思決定とマージ承認は必ず人間が実施します。

## 1. 参照優先順位

1. `docs/requirements.md`
2. `docs/specification.md`
3. `docs/team-rules.md`
4. `docs/docstring_ja_style.md`
5. `docs/skills.md`
6. `docs/llm-context/README.md`

## 2. エージェントへの要求

- PR前提の変更のみ提案すること。
- テスト・型・lintを通す変更を優先すること。
- 仕様変更時は関連ドキュメントを同時更新すること。
- 機密情報を生成物や提案に含めないこと。

## 3. 人間承認ルール

- AI生成コードは人間承認なしでマージしない（単独開発時は自己レビュー結果をPRへ記録）。
- 重大な設計判断はIssueまたはADR相当の記録を残す。

## 4. ワークフロー・オーケストレーション

### 4.1 Planノードの既定運用

- 非自明タスク（3ステップ以上、または設計判断を含む作業）はPlanモードから開始する。
- 想定外が起きた場合は作業を止め、即座に再計画する。
- Planモードは実装時だけでなく、検証工程にも適用する。
- 曖昧さを減らすため、着手前に詳細仕様を記述する。

### 4.2 サブエージェント戦略

- メインのコンテキストを保つため、サブエージェントを積極的に活用する。
- 調査、探索、並列分析はサブエージェントへ委譲する。
- 複雑な課題ではサブエージェントを増やして並列で解く。
- サブエージェントは1タスク1担当で実行させる。

### 4.3 自己改善ループ

- ユーザーから修正を受けたら毎回 `tasks/lessons.md` に再発防止パターンを追記する。
- 同じ失敗を防ぐための自分向けルールを明文化する。
- 失敗率が下がるまで lessons を継続的に改善する。
- セッション開始時に、対象プロジェクトに関連する lessons を確認する。

### 4.4 完了前の検証

- 動作証明なしにタスク完了としない。
- 必要に応じて `main` と変更後の挙動差分を確認する。
- 「Staff Engineerが承認できる品質か」を自己点検する。
- テスト、ログ確認、正しさの提示まで実施する。

### 4.5 エレガンス重視（バランス）

- 非自明な変更では「よりエレガントな方法がないか」を一度確認する。
- 場当たり的な修正になった場合は、より整った実装へ組み替える。
- 単純な修正では過剰設計を避ける。
- 提示前に自分の変更を批判的にレビューする。

### 4.6 自律的バグ修正

- バグ報告を受けたら、追加の手取り足取りを求めず修正まで進める。
- ログ、エラー、失敗テストを根拠に原因を解消する。
- ユーザーへ不要なコンテキストスイッチを求めない。
- 失敗中のCIは、手順指示を待たずに修正する。

## 5. タスク管理

1. **Plan First**: チェック可能な項目で計画を `tasks/todo.md` に記載する。
2. **Verify Plan**: 実装開始前に計画を確認する。
3. **Track Progress**: 進行に合わせて完了チェックを更新する。
4. **Explain Changes**: 各ステップで変更の要点を記録する。
5. **Document Results**: `tasks/todo.md` にレビュー結果セクションを残す。
6. **Capture Lessons**: 修正を受けたら `tasks/lessons.md` を更新する。

## 6. コア原則

- **Simplicity First**: 変更は可能な限り単純にし、影響範囲を最小化する。
- **No Laziness**: 根本原因を解消し、暫定対処で終わらせない。
- **Minimal Impact**: 必要箇所だけを変更し、不要なバグ混入を避ける。
