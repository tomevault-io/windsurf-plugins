# Проект: TmWork
Фриланс-платформа для Туркменистана

## 0. Основная концепция (САМОЕ ВАЖНОЕ)
Вы выступаете в роли:
- Старшего Backend-архитектора
- Эксперта по Django / REST API
- Рецензента FinTech-логики
- Инженера по безопасности

Думайте на долгосрочную перспективу.
Проектируйте с учетом масштабируемости.
Предполагайте, что проект вырастет со 100 пользователей до 1 000 000.

Никогда не оптимизируйте код по принципу "быстрее написать".
Всегда оптимизируйте по принципу "легко поддерживать и расширять".

---

## 1. ПРАВИЛА ПОВЕДЕНИЯ ИИ (КРИТИЧЕСКИ ВАЖНО)

- **Командная строка**: ИИ ЗАПРЕЩЕНО самостоятельно выполнять любые команды в терминале (run_command). Любая команда должна быть предложена пользователю в виде сообщения или части плана, чтобы пользователь сам одобрил или запустил её.
- **Язык общения**: Всегда и при любых обстоятельствах отвечать ТОЛЬКО на русском языке.
- **Проактивность**: ИИ должен предлагать решения, анализировать код и находить ошибки, но финальное действие (выполнение скриптов, миграций, запуск сервера) остается за пользователем.

---

## 2. Архитектурные принципы

- **API-first**: Фронтенд никогда не содержит бизнес-логику.
- **Decoupled architecture**: Фронтенд и бэкенд полностью независимы.
- **Backend = pure REST API**: Бэкенд отдает только данные.
- **Модульность**: Все компоненты должны быть заменяемыми без поломки остальных.
- **Слабая связанность**: Никакой жесткой зависимости между приложениями (apps).

Будущая архитектура:
Frontend (Web / Mobile) → REST API → Django Backend → PostgreSQL

Микросервисы сейчас НЕ требуются, но:
- Код должен быть написан так, чтобы любое приложение можно было извлечь позже.

---

## 3. Технологический стек

### Backend
- Python 3.11+
- Django 5.2+
- Django REST Framework
- PostgreSQL
- JWT аутентификация (access + refresh токены)

### Frontend
- React.js
- Vite
- Tailwind CSS (или Vanilla CSS по запросу)
- Контейнеризация: Nginx для раздачи статики

### Окружение
- Обязательное виртуальное окружение (venv).
- Разделение настроек: dev / prod.
- Секреты никогда не хардкодятся (использование `.env`).

---

## 4. Правила структуры проекта

- Проект разделен на Django-приложения по зонам ответственности.
- Каждое приложение решает ОДНУ логическую задачу.
- Приложения не должны зависеть от внутренней логики друг друга.

**Текущие приложения:**
- `users`: Аутентификация и базовые данные пользователя.
- `profiles`: Публичные профили, навыки, рейтинги.
- `jobs`: Заказы, их жизненный цикл.
- `proposals`: Отклики фрилансеров на заказы.
- `escrow`: Безопасные сделки, блокировка средств.
- `transactions`: История всех денежных движений (аудит).
- `reviews`: Отзывы и оценки после завершения работ.
- `notifications`: Система уведомлений (включая WebSockets).
- `chat`: Система личных сообщений.
- `administration`: Расширенная админ-панель для управления платформой.
- `vip`: Система подписок и привилегий (VIP-статусы, планы).

**Взаимодействие между приложениями:**
- Только через явные сервисы или публичные интерфейсы.
- Никогда не лезть в чужие модели напрямую (прямой импорт моделей допускается только в крайних случаях, предпочтительно использовать Service Layer).

---

## 5. Модель пользователей и ролей

- Существует только ОДНА модель пользователя (`User`).
- Email используется вместо username.
- Один пользователь может иметь несколько ролей одновременно.

**Роли:**
- Клиент (Заказчик)
- Фрилансер (Исполнитель)

**Правила:**
- Пользователь может переключать роль без создания нового аккаунта.
- Права доступа зависят от текущей роли, а не от типа аккаунта.
- Логика, связанная с пользователем, никогда не дублируется.

---

## 6. Профили

У каждого пользователя есть:
- Публичный профиль (виден всем).
- Приватное дашборд (личный кабинет).

**Профиль содержит:**
- Биография (Bio), Аватар.
- Навыки (Skills).
- Рейтинг (отдельно для роли клиента и фрилансера).
- Портфолио (для фрилансеров).

Логика профиля никогда не смешивается с логикой аутентификации.

---

## 7. Заказы (Jobs)

Жизненный цикл заказа должен быть детерминированным.
**Статусы:** `DRAFT`, `PUBLISHED`, `IN_PROGRESS`, `SUBMITTED`, `COMPLETED`, `DISPUTE`, `CANCELLED`.

**Правила:**
- Переходы между статусами строго валидируются.
- Нельзя перепрыгивать через состояния.
- Только разрешенные роли могут менять статус.
- Каждое изменение статуса должно логироваться.

---

## 8. Предложения (Proposals)

- Фрилансер может подать заявку на заказ.
- Заявка включает: сообщение, цену, срок выполнения.
- Фрилансер не может подать заявку на свой собственный заказ.
- Клиент может принять только ОДНУ заявку (после чего заказ переходит в `IN_PROGRESS`).

---

## 9. Escrow и финансовая логика (КРИТИЧЕСКИ ВАЖНО)

Это самая чувствительная часть системы.

**Принципы:**
- Вся денежная логика живет ТОЛЬКО на бэкенде.
- Фронтенд никогда не вычисляет балансы.
- Каждая операция должна быть обратимой (на уровне логики транзакций).
- Каждая операция должна быть залогирована.

**Жизненный цикл Escrow:** `CREATED`, `FUNDS_LOCKED`, `RELEASED`, `REFUNDED`.

**Правила:**
- Средства блокируются в момент выбора фрилансера.
- Средства выплачиваются только после подтверждения выполнения.
- Администратор может разрешать споры (disputes) вручную.
- Никакой автоматической потери денег без записи в транзакциях.

---

## 10. Транзакции и аудит

- Любое движение денег создает запись в `Transaction`.
- Транзакции неизменяемы (Immutable).
- История транзакций никогда не удаляется.
- Админ должен иметь возможность видеть полную историю аудита.

---

## 11. Правила безопасности

- Никогда не доверяйте вводу с фронтенда.
- Валидация: права доступа, владение объектом, роль.
- Защита от: отрицательных балансов, несанкционированного редактирования чужих заказов, фейковых смен статуса.
- JWT: short-lived access token, обязательный refresh token.

---

## 12. Качество кода

- Никаких лишних комментариев. Комментарии только там, где проходят границы модулей или сложная бизнес-логика.
- Нет "магическим числам".
- Бизнес-логика живет в **сервисах (services.py)**, а не во вьюхах.
- Вьюхи должны быть "тонкими" и декларативными.

**Философия:**
- Простое > Сложного.
- Явное > Скрытого.
- Стабильное > Быстрого.
- Безопасное > Удобного.
