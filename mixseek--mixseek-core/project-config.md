# AI Agent Instructions
このリポジトリで作業する全てのAIエージェント（Copilot, Cursor, Claude etc.）は、以下のガイドラインに従ってください。

## 0. プロジェクト概要
mixseekは、LLMを活用したマルチエージェントオーケストレーションフレームワークです。
リーダーエージェントと複数のメンバーエージェント構成を1チームとし、複数チームが指定された単一のタスクにコンペティション形式で取り組みます。
各チームはラウンドごとにタスクへの回答を提出し、提出物はEvaluatorによる評価を受けてスコア化され、リーダーボードに掲載されます。
各チームは評価フィードバックを受け、次のラウンドで回答を洗練させることを繰り返します。

## 1. 開発ガイドライン

### 共通ルール
- 開発者の共通語は日本語
    - コード内のコメント、Git コメント、ドキュメントなど基本的には日本語で記述。

### コーディングスタイルと命名規約
- Python バージョンや整形設定（`ruff`）は、ルートディレクトリの `pyproject.toml` に従う
- `src/` レイアウトのため、import はプロジェクト固有のパッケージ名を使用
- DRY原則に従う
- **ファイル・関数サイズ制限**: (※ `tests/` 配下は除く)
    - プログラムはできるだけ1ファイル 300 行以内に保ち、300 行を超える場合はファイル分割を検討する
    - Python の1関数はできるだけ 200 行以内に保ち、200 行を超える場合は関数の分割を検討する
- **設定管理**: 直接 `os.getenv` せず、プロジェクト共通の設定モジュール（例: `<project_pkg>.configs.config` 等）を介して型安全にアクセスする
- **ロギング**: プロジェクト共通のロガー（例: `<project_pkg>.configs.log.logger` 等）を使用し、構造化 JSON ログを出力する

### テスト指針
テスト駆動開発を**厳守**してください。

- フレームワーク: `pytest` + `pytest-cov`
- テスト検出: `test_*.py`, `Test*`, `test_*`
- カバレッジ目標: **80%以上**
- マーカー: `unit`, `integration`, `e2e`, `contract`, `performance` を適切に付与

### 品質指針

実装時は**makeによる品質チェックを常に実行**し、コード品質を維持してください。

- commit前に必ず以下のコマンドを実行し、必要な修正を行う
	- `make -C dockerfiles/ci lint format-check`
	- `make -C dockerfiles/ci type-check`
    - `make -C dockerfiles/ci test-fast`

### コミット & PR ガイドライン
- コミットメッセージは簡潔で具体的。必須プレフィックスはなし
- PR には概要・テスト結果（`make pipeline` など）・UI 変更時のスクリーンショットを添付
- 要件や仕様に影響する変更は `docs/` の該当資料へのリンクを記載
- エージェントが GitHub のコミット/issue/PR を作成する場合、本文の文末に `Created by 🤖<model_name>` を追加
  - 例:
    - Gemini の場合 `Created by 🤖Gemini`
- Conventional Commits 形式を**必須**とする

#### フォーマット

```
<type>(<scope>): <subject>

<body>

<footer>
```

#### 必須タイプ一覧

```
feat:     新機能の追加
fix:      バグ修正
docs:     ドキュメントのみの変更
refactor: リファクタリング（機能変更なし）
test:     テストの追加・修正
release:  バージョン更新・リリース（commitizenが使用）
```

### ドキュメント作成ガイドライン

- ドキュメントを保守する場合は、 `internal/sphinx.md` を参照してから作業を行う
- ドキュメントで作成する markdown (md) ファイルに関する方針
    - **1行はできるだけ 120 文字未満**に収めること
    - **1ファイルはできるだけ 500 行未満**に抑え、500 行を超える場合はファイル分割を検討する
    - **1セクションはできるだけ 100 行未満**に抑える

### セキュリティ & 設定の注意
- 開発は dev container 推奨。秘密情報は AWS Parameter Store で管理し、direnv 経由でメモリ上に展開する。機密情報はコミットしてはいけない
- 外部サービス（DB/LLM 連携）は、原則テストではモックし、必要時のみ integration/e2e を実行

## 2. メモリ管理（Claude Code 向け）

以下のタイミングで **ユーザの依頼を待たず、自律的に** メモリファイルを更新してください。

- **知見を得たとき**: 新しいアーキテクチャの決定、ハマったバグの解決策、プロジェクト固有のパターン
- **ユーザーの好みを確認したとき**: ツール選定、コードスタイル、ワークフローの好み
- **タスク完了後**: 作業内容がメモリに反映すべき変化を含む場合は完了時に更新する
- **セッション終了前**: 会話を終える前にメモリの更新漏れがないか確認する
