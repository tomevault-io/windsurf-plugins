## About

Kelvin is a compile-time dimensional analysis library written in Mojo, inspired by Rust's `uom` crate. It provides type-safe physical quantities where dimensional correctness is verified at compile time with zero runtime overhead.

## Commands

```bash
# Build the package
pixi run build              # mojo package kelvin -o kelvin.mojopkg

# Run all tests
pixi run test_all           # runs test, test_reject, and test_ctime

# Run unit tests (also regenerates tests and formats)
pixi run test

# Run only rejection tests (verify invalid code fails at compile time)
pixi run test_reject

# Format code (80-char line length)
pixi run format             # mojo format -l 80 kelvin test

# Run a single test file
pixi run mojo -D ASSERT=all -I . test/kelvin/test_<unit>.mojo
```

Tests run in parallel (max 4 workers) via `run_test.py`. Test files in `test/kelvin/` are auto-generated by `scripts/generate_tests.py` â€” do not edit generated files manually.

When asked about improving compile time peformance. uses the output of this command to measure it.

`pixi run test_ctime`

## Architecture

### Core Type System (`kelvin/quantity/`)

The library is built on three compile-time structs:

- **`Ratio[N, D]`** (`ratio.mojo`) â€” Compile-time rational number representing a scale factor (e.g., `Ratio[1000, 1]` for Kilo). Pre-defined SI prefixes: `Nano`, `Micro`, `Milli`, `Kilo`, `Mega`, `Giga`, etc.

- **`Dimension[Z, R, suffix]`** (`quantity.mojo`) â€” A single SI dimension with an exponent `Z` (e.g., `2` for squared), a `Ratio` scale `R`, and a string `suffix` (e.g., `"m"`, `"kg"`).

- **`Dimensions`** (`quantity.mojo`) â€” A bundle of 7 SI base dimensions (L, M, T, EC, TH, A, CD) plus an `Angle` component. These are compile-time parameters.

- **`Quantity[D, DT, Width]`** (`quantity.mojo`) â€” The main generic type:
  - `D`: `Dimensions` (compile-time parameter encoding the physical dimension)
  - `DT`: `DType` (numeric type, default `float64`)
  - `Width`: SIMD vector width (default `1`)

### Unit Modules (`kelvin/*.mojo`)

Each unit module (e.g., `time.mojo`, `length.mojo`) defines concrete `Quantity` aliases:

```mojo
alias Second = Quantity[Dimensions(L=..., T=Dimension[1, Ratio.Unitary, "s"]())]
alias Minute = Quantity[Dimensions(L=..., T=Dimension[1, Ratio[60, 1](), "min"]())]
```

Unit conversions use the `cast_from=` parameter which converts between quantities sharing the same dimensional structure but different scale ratios.

### Derived Units

Derived units (velocity, acceleration, energy, etc.) are created via dimension arithmetic on `Dimensions`. Multiplication adds exponents; division subtracts them. This is all resolved at compile time.

### Rejection Tests (`test/rejects/`)

These tests verify compile-time type enforcement â€” invalid operations (adding meters to seconds, mismatched scales, scalar ops on quantities, mismatched SIMD widths) must fail to compile. Run via `scripts/run_reject_tests.py` using `-D CASE=<n>` flags.

### Key Design Constraints

- `@always_inline("builtin")` is used pervasively; this means dimension operations at the `Dimension`/`Dimensions` level are **not** type-checked at compile time â€” only at the `Quantity` level.
- Integer `DType`s will truncate on scale conversion (e.g., converting 1 Kilometer to Meters as `int64` â†’ `1000`, but fractional conversions lose precision).
- The `proof/` environment (with `z3-solver`) can be used for mathematical verification of unit relationships.
