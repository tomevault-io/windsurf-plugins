## Commands

```bash
# Development workflows
make full-rebuild       # codegen + build Android libs — use after changing Rust signatures
make rust-only          # build Android libs only — use after changing Rust implementation
make android-dev        # full-rebuild + flutter run

# Individual steps
make codegen            # regenerate Dart bindings from Rust FFI (flutter_rust_bridge_codegen)
make build-android      # build arm64 native libs → android/app/src/main/jniLibs
make build-android-all  # build arm64 + armeabi-v7a + x86_64

# Quality
make fmt                # format Rust + Dart
make lint               # clippy (Rust) + dart analyze
make analyze            # dart analyze only
dart analyze --fatal-infos  # strict (used in CI)

# Tests
make test               # Flutter tests + Rust tests
flutter test path/to/test_file_test.dart  # single file
flutter test --coverage
make test-rust          # cargo test across all Rust crates

# Database
make migrate-new        # create a new SQLite migration with timestamp
make migrate-status     # list applied/pending migrations
```

Commits use Conventional Commits (`feat:`, `fix:`, `chore:`, etc.).

## Architecture

nav-e is a Flutter app with a Rust native engine. Flutter handles UI and state; Rust handles all business logic (routing, navigation, device communication, persistence).

### Rust workspace (`native/`)

Six crates, each with a single responsibility:

| Crate | Role |
|---|---|
| `nav_core` | All business logic — DDD/Hexagonal/CQRS, SQLite persistence |
| `nav_e_ffi` | Flutter-Rust bridge wrapper — `#[frb]`-annotated delegates only, no logic |
| `nav_protocol` | BLE/protobuf framing and chunking (aliased as `device_comm` in nav_core's Cargo.toml) |
| `nav_ir` | Canonical route intermediate representation shared across routing backends |
| `nav_route` | HTTP routing adapters (OSRM, Google Routes, Valhalla) |
| `nav_engine` | Turn-by-turn navigation engine (pure logic, depends only on `nav_ir`) |

`nav_core` is organised as **vertical bounded-context slices**: `navigation/`, `places/`, `devices/`, `offline/`, `shared/`. Each slice has `domain/`, `application/` (commands/queries/handlers), `infrastructure/`. `app/container.rs` holds the `AppContainer` singleton; `get_container()` is the **only** entry point from `api/`. The `api/` layer is thin wrappers — one `get_container().<context>.<method>(cmd)` call per function.

**Adding a Rust API end-to-end:**
1. Add command/query struct in `<context>/commands.rs` or `queries.rs`
2. Add handler in `<context>/handlers.rs`
3. Add thin wrapper in `native/nav_core/src/api/<feature>/`
4. Add 1-line `#[frb]` delegate in `native/nav_e_ffi/src/lib.rs`
5. `make codegen` → fix Dart call sites → `make test-rust`

### Flutter app (`lib/`)

- **FFI bridge**: `import 'package:nav_e/bridge/lib.dart' as api;` — this is the only import from `lib/bridge/`. Never edit `lib/bridge/` manually; it is fully generated by `make codegen`.
- **Repository implementations** wrapping FFI calls live in `lib/features/<feature>/data/*_repository_rust.dart`.
- **Global BLoCs** (`LocationBloc`, `BluetoothBloc`) live in `lib/core/bloc/` and are provided at app root in `main.dart` — do not re-register at feature level.
- **Feature BLoCs** (`search/`, `nav/`, `device_management/`, `plan_route/`) use full event/state split (`*_event.dart`, `*_state.dart`), `Equatable` states, handlers registered via `on<EventType>` in the constructor.
- **Cubits** (`saved_places/`, `offline_maps/`, `trip_history/`, `saved_routes/`) for simpler state.
- **Routing**: `lib/core/routing/` (GoRouter).

### Theme system

All styling comes from `lib/core/theme/` — never use `Colors.*`, raw `TextStyle`, or hardcoded spacing/shadow values in widget build methods.

- **Text**: `Theme.of(context).textTheme.*`
- **Colors**: `Theme.of(context).colorScheme.*` for Material tokens; `Theme.of(context).extension<AppColors>()!` for semantic tokens (`warning`, `info`, `disabled`, etc.)
- **Spacing**: `AppSpacing` constants (8-pt grid: `xs=4`, `sm=8`, `md=16`, `lg=24`, `xl=32`, `xxl=48`)
- **Shadows**: `AppElevation.level1–4(colorScheme.shadow)` from `elevation.dart`
- New component variants go in `lib/core/theme/components/` (follow the `badges.dart` pattern) and are registered in `app_theme.dart`.

### Device communication & protobuf

- `.proto` definitions live in `proto/`; compiled into Rust by `prost-build` in `native/nav_protocol/build.rs` and into Dart by the build system.
- After editing `.proto`: regenerate both Dart and Rust, and sync changes with nav-c (the companion device project).
- BLE frames: magic, msg_type, protocol_version, route_id, seq_no, total_seqs, payload_len, payload, crc32. Chunking is handled by `chunk_message_for_ble` in `native/nav_protocol/`.
- Flutter-side device comm service: `lib/core/device_comm/`.
- `nav_protocol` is aliased as `device_comm` in nav_core's Cargo.toml — `use device_comm::*` in Rust source is correct.
