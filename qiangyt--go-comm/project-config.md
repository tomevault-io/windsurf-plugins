# OpenSpec Instructions

These instructions are for AI assistants working in this project.

***所有会话、文档、代码注释都使用简体中文***

## TDD 强制规范

### 核心原则

1. **测试先行**：任何生产代码的编写，必须先有对应的失败测试
2. **最小实现**：只编写足以让测试通过的最小代码量
3. **持续重构**：测试通过后，立即重构代码，保持代码整洁
4. **使用不同的 sub agent 来编写测试和实现代码**
5. **全量回归**：每个 Phase 开始前和结束后都必须运行所有测试，确保整个系统测试通过

### 严格流程（Red-Green-Refactor）

#### 阶段一：Phase 前回归
- 运行所有测试，确保整个系统的所有测试都通过
- 如果有测试失败，必须先修复，确保从干净状态开始

#### 阶段二：Red（红灯）
- 编写一个失败的测试用例
- 运行测试，确认测试失败（红灯）
- **禁止**：在此阶段编写任何生产代码

#### 阶段三：Green（绿灯）
- 编写最小化的生产代码，仅使测试通过
- 运行测试，确认测试通过（绿灯）
- **禁止**：在此阶段进行代码重构或添加额外功能

#### 阶段四：Refactor（重构）
- 在测试保护下，优化代码结构
- 每次重构后，运行测试确保仍然通过
- **禁止**：在测试失败时进行重构

#### 阶段五：Phase 后回归
- 运行所有测试，不仅仅是当前相关的测试，而是整个系统的所有测试都必须通过
- 如果有测试失败（包括非本次修改导致的），必须修复

### Phase 级别的四步原则

每个 Phase（任务阶段）必须遵循以下四步：

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1: Phase 前回归                                        │
│  - 运行所有测试                  │
│  - 确保所有测试通过，覆盖率达标                              │
│  - 如果失败，先修复再继续                                    │
├─────────────────────────────────────────────────────────────┤
│  Step 2: Phase 内部开发（Red-Green-Refactor 循环）           │
│  - Red: 编写失败测试，运行测试确认失败                       │
│  - Green: 编写最小实现代码，仅使测试通过                     │
│  - Refactor: 在测试保护下重构代码，优化结构                  │
│  - 重复 Red-Green-Refactor 直至 Phase 内所有任务完成        │
├─────────────────────────────────────────────────────────────┤
│  Step 3: Phase 后回归                                        │
│  - 运行所有测试                  │
│  - 确保所有测试通过，覆盖率达标                              │
│  - 修复任何失败的测试（包括非本次修改导致的）                 │
├─────────────────────────────────────────────────────────────┤
│  Step 4: Phase 代码审查                                      │
│  - 运行 `superpowers:requesting-code-review` skill          │
│  - 处理审查反馈，修复问题                                    │
│  - 确保审查通过后进入下一 Phase                              │
│  - **更新 tasks.md 中的任务状态**（将 `- [ ]` 改为 `- [x]`） │
└─────────────────────────────────────────────────────────────┘
```

### Refactor 检查清单

每个 Task 完成后必须进行重构：
- [ ] 是否有重复代码需要提取？
- [ ] 函数是否过长需要拆分？
- [ ] 命名是否清晰准确？
- [ ] 是否有魔法数字需要常量化？
- [ ] 是否有可以简化的逻辑？

### 覆盖率要求

- **行覆盖率 (Line Coverage)**: 必须 = 100%
- **分支覆盖率 (Branch Coverage)**: 必须 = 100%
- 每个函数、每个分支、每个错误路径都必须有测试覆盖

### 禁止行为

- ❌ 跳过测试直接编写生产代码
- ❌ 先写生产代码后补测试
- ❌ 为了通过测试而跳过、注释或删除测试用例
- ❌ 提交覆盖率不达标的代码
- ❌ 在测试失败时进行代码重构
- ❌ 在一个测试用例中测试多个不相关的功能
- ❌ Phase 结束后留有失败的测试
- ❌ 忽略非本次修改导致的测试失败

### 验证检查点

每个 Phase 完成后，必须确认：
- [ ] 所有测试通过（全量回归）
- [ ] 行覆盖率 = 100%
- [ ] 分支覆盖率 = 100%
- [ ] 代码已通过重构优化
- [ ] 没有被注释或跳过的测试用例
- [ ] 非本次修改导致的问题也已修复
- [ ] 代码审查通过（`superpowers:requesting-code-review`）
- [ ] **更新 tasks.md 中的任务状态**（将 `- [ ]` 改为 `- [x]`）

## 基本规则

1. 任何场合（包括代码注释）、除了 UI 界面要支持国际化，其它都使用中文
2. 当我报告一个 Bug 时，不要急着修复。先写一个能复现这个 Bug 的测试，然后让子代理去修复，并用通过的测试来证明修复有效

## Error 处理规范

### 核心原则

1. **禁止使用 error 返回值**：所有函数都不应该返回 error
2. **使用 panic 替代 error**：任何错误情况都使用 panic
3. **仅在 main 和 goroutine 最外层捕获 panic**：
   - `main()` 函数必须有 defer recover
   - 每个 goroutine 的最外层必须有 defer recover

### 具体规则

1. **函数签名**：移除所有 error 返回值
   - ❌ `func Foo() (Result, error)`
   - ✅ `func Foo() Result`

2. **错误处理**：使用 panic 替代 return error
   - ❌ `if err != nil { return nil, err }`
   - ✅ `if err != nil { panic(fmt.Errorf("操作失败: %w", err)) }`

3. **main 函数**：必须添加 panic recovery
   ```go
   func main() {
       defer func() {
           if r := recover(); r != nil {
               fmt.Fprintf(os.Stderr, "panic: %v\n", r)
               os.Exit(1)
           }
       }()
       // ...
   }
   ```

4. **goroutine**：必须在最外层添加 panic recovery
