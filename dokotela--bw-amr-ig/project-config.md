## Project Overview

FHIR Implementation Guide (IG) for the **Botswana AMR (Antimicrobial Resistance) Data Analytics Platform**. Uses FHIR R4 (v4.3.0), authored in FSH (FHIR Shorthand), and includes a companion Flutter/Dart package for consuming the generated artifacts. Designed for WHONET export compatibility and GLASS reporting alignment.

## Common Commands

### Prerequisites
```bash
sudo npm install -g fsh-sushi
# Java required for IG Publisher (publisher.jar)
```

### Build the IG
```bash
# Compile FSH to FHIR JSON resources
sushi .

# Regenerate ConceptMap resources from mapping CSVs
python3 python/generate_concept_maps.py

# Full IG build (runs publisher.jar, copies artifacts to flutter/, generates Dart files, serves output)
./_genonce.sh
```

### Flutter package
```bash
cd flutter
dart pub get
dart analyze
dart test
```

### Export package
```bash
cd export
dart pub get
dart test
```

## Architecture

### FSH Source (`input/fsh/`)

All FHIR conformance resources are authored in FSH:

- **Profiles** (`profiles/`): 8 profiles constraining base FHIR resources for AMR workflows ŌĆö `DiagnosticReport`, `Patient`, `Specimen`, `Encounter`, and four `Observation` types (GramStain, Organism, Susceptibility, SpecialTest)
- **ValueSets** (`valueSets/`): Bind coded fields to LOINC, SNOMED, and local codes (organisms, antibiotics, specimen types, interpretation, AST methods, ward types, etc.)
- **CodeSystems** (`codeSystems/`): Local code systems for organisms and identification methods not in standard terminologies
- **Extensions** (`extensions/`): Custom extensions (e.g., specimen collection location)

### Resource Hierarchy

```
DiagnosticReport (top-level, references Encounter for ward/admission context)

Ōö£ŌöĆŌöĆ GramStainObservation (gram stain result)
ŌööŌöĆŌöĆ OrganismObservation (identified organism, isolate number)

    ŌööŌöĆŌöĆ SpecialTestObservation (ESBL, carbapenemase, MRSA, beta-lactamase, D-zone)
```

Key design: OrganismObservation uses `hasMember` to group its susceptibility and special test results. This properly handles multi-organism isolates (each organism links to its own AST panel).

### ConceptMaps (`concept-maps/`)

FHIR ConceptMap resources bridging WHONET codes to standard terminologies. Stored in `concept-maps/` (not `input/resources/`) to avoid IG Publisher attempting to validate ~5000 organism mappings against the terminology server, which causes it to hang.

- `ConceptMap-whonet-organism-to-snomed.json` ŌĆö ~5000 WHONET organism codes ŌåÆ SNOMED CT
- `ConceptMap-whonet-antibiotic-to-loinc.json` ŌĆö ~271 WHONET antibiotic codes ŌåÆ LOINC susceptibility codes
- `ConceptMap-whonet-antibiotic-to-atc.json` ŌĆö ~374 WHONET antibiotic codes ŌåÆ WHO ATC classification
- `ConceptMap-whonet-antibiotic-to-snomed.json` ŌĆö 42 key WHONET antibiotic codes ŌåÆ SNOMED CT substance codes

Generated by `python/generate_concept_maps.py` from CSV source data in `python/`.

### Build Pipeline (`_genonce.sh`)

1. Runs `publisher.jar` to compile the IG (FSH ŌåÆ JSON resources ŌåÆ HTML)
2. Copies generated CodeSystem/ValueSet JSON into `flutter/assets/`
3. Converts each JSON artifact to a Dart file (`flutter/lib/fhir/`) using a Python helper, creating typed `fhir_r4` objects

### Flutter Package (`flutter/`)

A Dart library (not a Flutter app) that wraps the IG artifacts for use in Dart/Flutter applications:

- `lib/fhir/` ŌĆö Auto-generated Dart files with pre-parsed `CodeSystem` and `ValueSet` objects (do not edit manually)
- `lib/models/` ŌĆö Hand-written model classes for each profile (Patient, Specimen, Observations, DiagnosticReport)
- `lib/value_set_extension.dart` ŌĆö Extension on `ValueSet` for working with coded values
- Depends on `fhir_r4` package

### Export Package (`export/`)

Standalone Dart package (`bw_amr_export`) for generating WHONET and GLASS flat-file exports from FHIR resources:

- `lib/src/concept_map_index.dart` ŌĆö Bidirectional index from ConceptMap JSON; builds reverse lookups (LOINCŌåÆWHONET, SNOMEDŌåÆWHONET, WHONETŌåÆATC)
- `lib/src/resource_resolver.dart` ŌĆö Walks FHIR resource graph (DiagnosticReportŌåÆOrganismŌåÆSusceptibility) to produce flat `IsolateRow` objects
- `lib/src/isolate_row.dart` ŌĆö Data model for one isolate row with demographics, specimen, organism, and antibiotic results
- `lib/src/whonet_exporter.dart` ŌĆö Pipe-delimited WHONET format output
- `lib/src/glass_exporter.dart` ŌĆö CSV output with ATC codes and HA/CO origin classification for WHO GLASS
- Depends on `fhir_r4: ^0.5.1`; uses `.valueString` (not `.value`) for primitive access

## Key Domain Concepts

AMR testing workflow modeled as FHIR resources:
- A **DiagnosticReport** groups all results for a specimen, linked to an **Encounter** for ward type and admission context
- **Encounter** captures ward type (inpatient/outpatient/ICU), admission date (for GLASS hospital-acquired classification), and facility
- **Specimen** captures sample type, collection site, and collection date
- **GramStainObservation** records initial gram stain classification
- **OrganismObservation** identifies the cultured organism (SNOMED-coded), with isolate number and `hasMember` links to child observations
- **SusceptibilityObservation** records antibiotic sensitivity (LOINC-coded, S/I/R interpretation with CLSI/EUCAST version tracking, MIC or disk diffusion zone diameter, AST method)
- **SpecialTestObservation** captures additional tests (ESBL, carbapenemase, MRSA, beta-lactamase, D-zone)

All Observation profiles include mandatory `category` codings (laboratory + microbiology studies) and `dataAbsentReason` invariants.

## Design Decisions

- **Aligned with German MII microbiology module** for `hasMember` grouping pattern, `category` coding, interpretation version tracking, and UCUM bindings
- **WHONET export-oriented**: profiles capture all WHONET-required fields (ward type, admission date, AST method, disk diffusion zone size, isolate number)
- **Terminology strategy**: LOINC for observation codes, SNOMED CT for organisms and substances, ATC for WHO GLASS alignment, local CodeSystems for Botswana-specific codes
