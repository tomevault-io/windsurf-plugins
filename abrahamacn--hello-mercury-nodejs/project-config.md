## Commands

- `npm run build` ‚Äî `tsc -p tsconfig.json` then `node copy-resource-files.js` (copies `src/resources/` ‚Üí `dist/resources/`). The copy step is required; without it the app aborts at startup with `Missing resources folder`.
- `npm start` ‚Äî runs `node dist/main.js` (must build first).
- No test runner or linter is configured.

The `mercury-composable` dependency is a `file:../mercury-nodejs` link. That sibling library must be built (`npm install && npm run build` in `../mercury-nodejs`) before this project will resolve its imports.

Default REST port is `8084`. Endpoints:
- `GET /api/hello/direct/{name}` ‚Äî Style 1 (direct)
- `GET /api/hello/flow/{user}` ‚Äî Style 2 (flow)

## Architecture

This is a minimal Mercury Composable app intentionally mirroring the sibling `hello-world-java` project so the two can be compared 1:1. It demonstrates the **two ways** Mercury wires a REST request to a user function:

1. **Direct style** ‚Äî `rest.yaml` routes the HTTP request straight to the `hello.direct` function (`src/services/hello-direct.ts`). The function receives the raw `AsyncHttpRequest` and pulls path params itself.
2. **Flow style** ‚Äî `rest.yaml` routes to the built-in `http.flow.adapter`, which dispatches into `resources/flows/greetings.yml`. The YAML flow extracts `input.path_parameter.user` into a `user` field, then invokes `greeting.demo` (`src/services/greeting-demo.ts`), which just reads from its input map.

### Bootstrap path

`main.ts` ‚Üí `ComposableLoader.initialize()` performs the wiring in a fixed order:

1. Locate `dist/resources/` and load `application.yml` via `AppConfig.getInstance(resourcePath)`.
2. Register user functions on the `Platform` singleton (`hello.direct`, `greeting.demo`), each with 10 worker instances.
3. Start the `EventScriptEngine` ‚Äî this is what compiles the YAML flows under `resources/flows/`.
4. If `rest.automation=true` in config, start `RestAutomation` (the built-in Express server that reads `rest.yaml`).
5. `platform.runForever()` + `await platform.getReady()` keeps the process alive.

`composable-loader.ts` is **hand-written on purpose** so the wiring is visible. In a real Mercury app this file is generated by a class scanner driven by `src/resources/templates/preload.template` shipped with the library. If you're tempted to "improve" the loader by hiding things, don't ‚Äî the verbosity is the point.

### Resources

Everything under `src/resources/` is copied verbatim into `dist/resources/` at build time and is what `AppConfig` reads at runtime:
- `application.yml` ‚Äî top-level config (port, `rest.automation`, `yaml.flow.automation`, `static.html.folder`, etc.)
- `rest.yaml` ‚Äî REST endpoint ‚Üí function mappings
- `flows.yaml` + `flows/greetings.yml` ‚Äî flow definitions consumed by `EventScriptEngine`
- `public/` ‚Äî static assets for `RestAutomation`'s `express.static('classpath:/public')`. Must exist (even if empty) or Express throws at startup.

### Gotchas worth knowing

- Functions are registered by **string name** (`'hello.direct'`, `'greeting.demo'`); these strings must match what `rest.yaml` and the flow YAML reference.
- The `@preload` decorator on each `Composable` is descriptive only here ‚Äî actual registration happens explicitly in `ComposableLoader`.
- A 404 on `/api/hello/flow/...` usually means `flows.yaml` or `flows/greetings.yml` didn't make it into `dist/resources/`, or `yaml.flow.automation` is missing from `application.yml`.
