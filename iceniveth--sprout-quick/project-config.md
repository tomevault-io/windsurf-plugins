# Copilot Instructions — sprout-quick

## Project Overview

**sprout-quick** is a full-stack React application that automates clocking in/out on Sprout HR. It is built with **React Router v7** (framework mode, SSR enabled) running on a **Cloudflare Worker** via **Wrangler**. Styling uses **Tailwind CSS v4**.

## Tech Stack & Versions

| Layer     | Technology                    | Version        |
| --------- | ----------------------------- | -------------- |
| Framework | React Router (framework mode) | ^7.10          |
| UI        | React                         | ^19.1          |
| Runtime   | Cloudflare Workers            | wrangler ^4.71 |
| Styling   | Tailwind CSS (Vite plugin)    | ^4.1           |
| Language  | TypeScript (strict)           | ^5.9           |
| Bundler   | Vite                          | ^7.1           |
| Linter    | oxlint                        | ^1.51          |
| Formatter | Prettier                      | 3.8            |

## Project Structure

```
app/                  # React Router application code
  app.css             # Global CSS — imports Tailwind via @import "tailwindcss"
  entry.server.tsx    # SSR entry — renders to ReadableStream for Workers
  root.tsx            # Root layout, meta, error boundary
  routes.ts           # Route config using @react-router/dev/routes helpers
  routes/             # Route modules (file-based convention within this folder)
    home.tsx          # Index route — each route module may export loader, action, meta, default component
public/               # Static assets (favicon, etc.)
workers/
  app.ts              # Cloudflare Worker entry point — bridges Worker fetch to React Router
worker-configuration.d.ts  # Auto-generated by `wrangler types` — DO NOT edit manually
.react-router/        # Auto-generated types by `react-router typegen` — gitignored
```

## Key Architecture Patterns

### React Router v7 (Framework Mode)

- Routes are defined in `app/routes.ts` using helper functions (`index`, `route`, `prefix`, `layout`) from `@react-router/dev/routes`.
- Each route module can export: `loader` (GET data), `action` (mutations), `meta`, `links`, `headers`, `handle`, `ErrorBoundary`, and a `default` component.
- The default component receives `{ loaderData, actionData, params, matches }` typed via `Route.ComponentProps`.
- Loaders and actions receive `{ request, params, context }` where `context` provides Cloudflare bindings.
- Type-safe route types are auto-generated at `.react-router/types/` — import them as `import type { Route } from "./+types/<routename>"`.
- SSR is enabled (`ssr: true` in `react-router.config.ts`). The `future.v8_viteEnvironmentApi` flag is on.
- Use `react-router` imports (not `@remix-run/*`) — this project uses React Router v7, NOT Remix.

### Cloudflare Workers Integration

- The Worker entry is `workers/app.ts`. It creates a request handler and passes `cloudflare: { env, ctx }` as the app load context.
- Access Cloudflare bindings (KV, D1, R2, AI, env vars, etc.) inside loaders/actions via `context.cloudflare.env`.
- The `Env` interface is declared in `worker-configuration.d.ts`, auto-generated by `wrangler types` (`npm run cf-typegen`). After adding bindings in `wrangler.jsonc`, always run `npm run cf-typegen` to regenerate types.
- The `AppLoadContext` interface is augmented in `workers/app.ts` to include `cloudflare: { env: Env; ctx: ExecutionContext }`.
- Wrangler config is in `wrangler.jsonc` (JSON with comments). `compatibility_flags: ["nodejs_compat"]` is enabled.
- Vite uses `@cloudflare/vite-plugin` with `viteEnvironment: { name: "ssr" }`.

### Styling

- Tailwind CSS v4 with the Vite plugin (`@tailwindcss/vite`). No `postcss.config` needed.
- Global CSS is in `app/app.css` — uses `@import "tailwindcss" source(".")` and `@theme` for custom design tokens.
- A legacy `tailwind.config.js` exists for reference; the primary alias is `primary → orange`.
- Prefer Tailwind utility classes directly in JSX. Use the `primary-*` color alias for brand colors.

### TypeScript

- Strict mode is enabled across all tsconfigs.
- `verbatimModuleSyntax` is on — always use `import type` for type-only imports.
- Path alias: `~/*` maps to `./app/*` (e.g., `import { foo } from "~/utils/foo"`).
- Two tsconfig references: `tsconfig.node.json` (Vite config) and `tsconfig.cloudflare.json` (app + workers).

## Coding Conventions

### General

- Use **TypeScript** for all files. No plain `.js`/`.jsx` in `app/` or `workers/`.
- Use **ES modules** (`"type": "module"` in package.json). No CommonJS in app or worker code.
- Use `satisfies` for type-safe config objects where applicable.
- Prefer `const` over `let`. Never use `var`.
- Use template literals over string concatenation.

### React & Components

- Use **function declarations** or **arrow function expressions** for components (the codebase uses both — either is acceptable, stay consistent within a file).
- Always type component props explicitly when they are non-trivial.
- Use React 19 features where appropriate (use, server components awareness).
- Keep components in `app/routes/` for route modules. Shared UI components should go in `app/components/`.
- Avoid default exports except for route module components (required by React Router).

### Route Modules

When creating a new route:

1. Add the route in `app/routes.ts` using helpers: `index()`, `route(path, file)`, `layout()`, `prefix()`.
2. Create the route file in `app/routes/`.
3. Import types from the auto-generated `./+types/<filename>` module.
4. Export a `meta` function for title/description.
5. Export a `loader` function for server data fetching. Access Cloudflare bindings via `context.cloudflare.env`.
6. Export an `action` function for form mutations.
7. Export a `default` component receiving `Route.ComponentProps`.

Example route module pattern:

```tsx
import type { Route } from "./+types/my-route";

export function meta({}: Route.MetaArgs) {
  return [{ title: "Page Title" }, { name: "description", content: "..." }];
}

export async function loader({ context, request }: Route.LoaderArgs) {
  // Access Cloudflare bindings
  const value = context.cloudflare.env.MY_BINDING;
  return { value };
}

export async function action({ context, request }: Route.ActionArgs) {
  const formData = await request.formData();
  // process mutation
  return { success: true };
}

export default function MyRoute({ loaderData }: Route.ComponentProps) {
  return <div>{loaderData.value}</div>;
}
```

### Adding Cloudflare Bindings

When adding new bindings (KV, D1, R2, Queues, Durable Objects, AI, env vars, etc.):

1. Add the binding in `wrangler.jsonc`.
2. Run `npm run cf-typegen` to regenerate `worker-configuration.d.ts`.
3. Access it in loaders/actions via `context.cloudflare.env.BINDING_NAME`.
4. For local dev secrets, add them to `.dev.vars` (gitignored).

### Error Handling

- Route-level error boundaries are supported. Export an `ErrorBoundary` component from route modules.
- The root error boundary in `root.tsx` catches unhandled errors globally.
- Use `isRouteErrorResponse(error)` to differentiate expected HTTP errors from unexpected ones.

## Important Constraints

- **Do NOT edit** `worker-configuration.d.ts` — it is auto-generated.
- **Do NOT edit** files in `.react-router/` — they are auto-generated.
- After modifying `wrangler.jsonc` bindings, always run `npm run cf-typegen`.
- After adding/renaming routes, run `npx react-router typegen` (or `npm run typecheck`) to regenerate route types.
- This runs on Cloudflare Workers — **no Node.js APIs** unless polyfilled via `nodejs_compat`. Prefer Web Standard APIs (fetch, Request, Response, URL, crypto, streams, etc.).
- Keep bundle size small — Workers have a 10 MB compressed limit for free tier, 25 MB for paid.
- Use `import type` for type-only imports (enforced by `verbatimModuleSyntax`).
