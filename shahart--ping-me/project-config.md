## Build & Run Commands

Gradle is preferred over Maven.

```bash
# Build (skip tests)
./gradlew build -x test

# Run application
./gradlew bootRun

# Run all tests
./gradlew test

# Run a single test class
./gradlew test --tests "edu.pingme.messaging_stomp_websocket.UtilsTest"

# Run a single test method
./gradlew test --tests "edu.pingme.messaging_stomp_websocket.UtilsTest.testDeCrypt"

# Package as WAR
./gradlew assemble
```

## External Prerequisites

Before running, Chrome must be started with remote debugging enabled:
```bash
chrome --remote-debugging-port=9222
```

The application connects to Chrome at `127.0.0.1:9222` via Selenium to automate WhatsApp Web. ChromeDriver must be installed and match the Chrome-for-Testing version.

For smartwatch access, expose the server with ngrok:
```bash
ngrok http 8080
```

## Architecture

**Single-controller REST app.** The entire backend logic lives in `Controller.java`:
- `POST /ping/{recipient}` is the only active endpoint
- On startup (`@PostConstruct`), instantiates a ChromeDriver connecting to the running Chrome session and runs a sanity check by sending a test message
- Selenium automation finds the WhatsApp message input box by CSS class (configured as `recipientElementId` in `application.properties`) and types/sends messages
- Messages are XOR-encrypted on the frontend (JS in `index.html`) and decrypted in the controller before being sent to WhatsApp

**Encryption:** Symmetric XOR cipher with Base64 encoding. Both `Utils.deEnCrypt()` (backend) and the inline JS in `index.html` implement the same algorithm â€” they must stay in sync.

**Predefined messages:** Six message templates are hardcoded in `Controller.java` (lines matching `^0$` through `^5$` prefix patterns). Two plain-text presets (`message.1`, `message.2`) come from `application.properties`.

**Auto-shutdown:** `MessagingStompWebsocketApplication.java` schedules a check every 5 minutes; if `Controller.getLastUsage()` shows no activity for 6 hours, the JVM exits.

**Frontend:** `src/main/resources/static/index.html` is a single-page UI optimized for small screens (iWatch). No build pipeline â€” plain HTML/JS served as Spring Boot static resources. The emoji dictionary page (`emojies.html`) is generated by the `@Disabled` test `UtilsTest.generateDictionary()`.

## Configuration

`src/main/resources/application.properties`:
- `password` â€” default `1234`, validated client-side and used in XOR encryption key
- `recipient.source` / `recipient.target` â€” WhatsApp contact name to select
- `recipientElementId` â€” CSS class selector for the WhatsApp Web message input box (fragile; breaks when WhatsApp updates its DOM)
- `message.1`, `message.2` â€” predefined message text

## Test Notes

`UtilsTest` tests are pure unit tests and run without Chrome. `MessagingStompWebsocketApplicationTests.chromeIsUp()` requires a running Chrome on port 9222; it will fail in CI (CI runs with `-x test` / `-DskipTests=true` for this reason).

The `generateDictionary()` test is `@Disabled` and is only meant to be run manually to regenerate the emoji HTML page.
