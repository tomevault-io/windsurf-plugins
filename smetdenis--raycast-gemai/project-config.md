# Git Workflow и Коммиты

## Политика Git коммитов

### 🚫 КРИТИЧЕСКИЕ ОГРАНИЧЕНИЯ
- **КРИТИЧНО**: Коммиты делать ТОЛЬКО если пользователь ЯВНО попросил
- **НИКОГДА** не делать коммиты самостоятельно без ЯВНОГО запроса пользователя
- **🚫 НИКОГДА НЕ ДЕЛАТЬ `git push` БЕЗ ЯВНОЙ КОМАНДЫ ПОЛЬЗОВАТЕЛЯ!**
- **ЗАПРЕЩЕНО** использовать `git push` без прямой команды пользователя
- **ОБЯЗАТЕЛЬНО** спрашивать разрешение перед любыми Git операциями
- **ТОЛЬКО РУССКИЙ ЯЗЫК** в сообщениях коммитов - никаких английских сообщений

### Формат коммитов

#### Структура сообщения
```
тип: краткое описание изменений

Детальное описание что именно изменено:
- Конкретные файлы и их изменения
- Причина изменений
- Влияние на функциональность
```

**Детальные коммиты**: Указывай что именно изменено и в каких файлах
Формат коммитов: `тип: детальное описание изменений`

#### Типы коммитов
- `feat:` - новая функциональность или возможность
- `fix:` - исправление багов и ошибок
- `refactor:` - изменение кода без изменения функциональности (рефакторинг без изменения API)
- `docs:` - изменения в документации
- `style:` - форматирование, исправление стиля кода (пропущенные точки с запятой и т.д.)
- `test:` - добавление или изменение тестов
- `chore:` - обновление зависимостей, настройки проекта (настройки сборки)
- `perf:` - улучшение производительности
- `build:` - изменения в системе сборки
- `ci:` - изменения в конфигурации CI/CD

### Примеры коммитов

#### Правильные коммиты из практики проекта
```bash
feat: добавлена поддержка GPT-4o в buildOpenAIConfig.ts и обновлены цены в models.ts
fix: исправлена ошибка подсчёта токенов в aiProvider.ts для reasoning моделей
refactor: переход на универсальный AIConfig в gemai.tsx, buildAIConfig.ts
docs: обновлена документация API провайдеров в README_RUS.md
style: форматирование кода в src/core/ согласно prettier
test: добавлены тесты для функций подсчёта токенов в utils.test.ts
```

#### Расширенные примеры с детальным описанием
```bash
feat: добавлена поддержка модели GPT-4o в OpenAI провайдере

Изменения:
- Обновлен buildOpenAIConfig.ts с новой моделью
- Добавлены цены для GPT-4o в models.ts
- Обновлена типизация в types.ts
```

```bash
fix: исправлена ошибка подсчёта токенов для reasoning моделей

Изменения:
- aiProvider.ts: правильная обработка reasoning tokens для o1 моделей
- utils.ts: добавлена функция для расчёта стоимости thinking tokens
- Исправлен баг с неточным подсчётом общей стоимости запроса
```

```bash
refactor: унификация конфигурации AI провайдеров

Изменения:
- Создан единый интерфейс AIConfig в types.ts
- Обновлены buildGemAIConfig.ts и buildOpenAIConfig.ts
- gemai.tsx теперь использует универсальный buildAIConfig()
- Упрощена логика переключения между провайдерами
```

#### Неправильные коммиты (избегать)
```bash
# ❌ Английский язык
feat: add GPT-4o support

# ❌ Слишком общие
fix: bug fixes

# ❌ Без деталей
update: code changes

# ❌ Неописательные
refactor: improvements
```

### Workflow для разработки

#### Перед коммитом
1. **Проверить изменения**: `git status` и `git diff`
2. **Убедиться в качестве**: запустить линтер и тесты
3. **Сформулировать сообщение**: детально описать что и зачем изменено
4. **Получить разрешение**: спросить у пользователя на коммит

#### Процесс коммита
1. Добавить файлы: `git add .` (только если пользователь согласен)
2. Создать коммит с детальным сообщением на русском
3. **НЕ ДЕЛАТЬ `git push`** без явной команды пользователя

#### Работа с ветками
- Использовать feature branches для новых возможностей
- Именовать ветки описательно: `feature/gpt4o-support`, `fix/token-counting`
- Делать rebase вместо merge для чистой истории (только с разрешения)

### Правила для AI ассистента

#### При работе с кодом
1. **Всегда спрашивать** перед Git операциями
2. **Предупреждать** о потенциальных коммитах
3. **Показывать diff** перед коммитом
4. **Объяснять изменения** детально

#### Примеры диалога
```
AI: Я внёс изменения в следующие файлы:
- src/core/models.ts (добавлена модель GPT-4o)
- src/core/buildOpenAIConfig.ts (обновлена логика)

Хотите ли вы сделать коммит с этими изменениями?
Предлагаемое сообщение:
"feat: добавлена поддержка GPT-4o в OpenAI провайдере"
```

#### Запрещённые действия
- Автоматические коммиты без разрешения
- Push в удалённый репозиторий без команды
- Изменение истории коммитов без разрешения
- Использование английского языка в сообщениях

### Исключения

#### Когда можно предложить коммит
- После завершения большой задачи
- При критических исправлениях
- По явной просьбе пользователя организовать коммиты

#### Как предлагать
```
Предлагаю сделать коммит с изменениями:

Файлы:
- src/core/aiProvider.ts
- src/core/types.ts

Сообщение:
"fix: исправлена обработка ошибок API в провайдерах"

Согласны? (да/нет)
```

## Версионирование и теги

### Semantic Versioning
- Используйте семантическое версионирование (MAJOR.MINOR.PATCH)
- MAJOR: Breaking changes в API
- MINOR: Новая функциональность (обратно совместимая)
- PATCH: Исправления ошибок

### Теги релизов
- Создавайте теги для релизов: `v1.2.3`
- Включайте changelog в описание релиза
- Документируйте breaking changes
