### Database Tables

- `users` - User accounts (email, name, activation_status, payment_status, password_hash)
- `user_packages` - Purchased packages (user_email, package_type, remaining_sessions, status)
- `reservations` - Booked sessions (user_email, date_key, time_slot, reservation_status)
- `redemption_codes` - Bonus/promo codes
- `kv_store_b87b0c07` - KV store (**sessions only** - all domain data migrated to Postgres)

### Session Management

- User sessions: 30-day sliding expiration (stored in KV)
- Admin sessions: 24-hour sliding expiration (stored in KV)
- Sessions extend on every authenticated request
- Frontend stores token in localStorage (`wellnest_session` for users, `adminSessionToken` for admin)
- Sent to backend via `X-Session-Token` header

## Critical: Timezone Handling

**All date/time logic uses Europe/Skopje timezone.** Never use `new Date()` directly for business logic.

Use centralized utilities from `dateUtils.ts`:
- `getSkopjeTime()` - Current time in Skopje
- `getSkopjeToday()` - Today at midnight in Skopje
- `isValidBookingDate(date)` - Checks not-past + weekday
- `formatDateKey(date)` - ISO format "2026-02-03"
- `formatDateKeyLegacy(date)` - Legacy "2-3" (month-day)
- `isTimeSlotPast(date, time)` - Whether slot has passed (5min buffer)

**Both files must stay in sync.** When updating either:
1. Make identical changes to `src/utils/dateUtils.ts` and `supabase/functions/make-server-b87b0c07/dateUtils.ts`
2. Update "Last synced" comment at top of both files

Constants defined in `dateUtils.ts`:
```
TIME_SLOTS = ['09:00', '10:00', '11:00', '17:00', '18:00', '19:00', '20:00']
MAX_CAPACITY = 4
PACKAGE_VALIDITY_DAYS = 35
```

## Key Business Logic

### Cancellation Rules
- 24+ hours before class: Can cancel (session refunded)
- Within 24 hours: Only within 2-minute grace period after booking
- No-show: Session consumed as penalty

### Package Flow
1. User selects package â†’ Creates pending package
2. Books first session â†’ Links reservation to package
3. Admin activates (after cash payment in studio) â†’ Package active, sends login email with magic link
4. User sets password via magic link â†’ Can now log in
5. Subsequent sessions â†’ Auto-confirmed, no activation needed

### Booking Capacity
Reservations use a Postgres atomic RPC with `FOR UPDATE` lock to prevent overbooking. Each time slot has `MAX_CAPACITY = 4` spots.
