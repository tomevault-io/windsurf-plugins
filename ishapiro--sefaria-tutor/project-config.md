# D1 Database & Migration Conventions

- **Runtime access**
  - Application code MUST use the native D1 API (`env.DB.prepare(...)`) and NOT import `drizzle-orm` in Worker code.
  - Keep all Drizzle usage in dev tooling (CLI) and TypeScript schema files.

- **Migrations directory**
  - Treat `migrations/` as the single source of truth for schema changes.
  - New migrations SHOULD be generated by Drizzle Kit and then reviewed/edited, but must remain plain SQL.

- **Drizzle Kit usage**
  - Use Drizzle Kit as a **headless migration generator** only.
  - `drizzle.config.ts` should:
    - Point `schema` to a TypeScript schema file (e.g. `./drizzle/schema.ts`).
    - Point `out` to `./migrations`.
    - Use the `sqlite` dialect and a SQLite URL compatible with the current D1 schema.
  - Do NOT wire Drizzle ORM directly into request handlers or runtime DB calls.

- **Baseline / sync workflow**
  - When the live D1 schema is out of sync, first align `schema.ts` with the **current** database (either by introspection or by copying from `1000_full_schema.sql` and existing migrations).
  - After updating `schema.ts`, run `drizzle-kit generate` and ensure it produces either no changes or only additive/expected SQL. Avoid destructive auto-generated changes without explicit review.

- **Applying migrations**
  - Always apply SQL migrations using Wrangler / the existing migration wrapper scripts (e.g. `npm run d1:migrate:local` / `npm run d1:migrate:remote`).
  - Do NOT introduce a second migration runner; Drizzle is for **generation**, Wrangler is for **execution**.
