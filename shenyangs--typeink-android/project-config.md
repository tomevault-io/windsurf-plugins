# 项目协作规则

本项目默认使用中文沟通。面向用户的解释，优先讲“这是什么、为什么这样做、会带来什么影响”。

## 强制维护文档

任何接手本项目的 AI 编程助理，都必须保持以下习惯：

1. 开始动手前，先阅读 [docs/系统架构与进度.md](/Users/sam/Desktop/typeink-codex/docs/系统架构与进度.md)。
2. 在开始当前任务前，先把你对系统架构的判断、当前目标、约束和计划补进该文档。
3. 完成当前任务后，必须把最新进度、变更内容、风险、下一步建议补进该文档。
4. 如果你改变了技术路线、目录结构、协议、状态机、模型接入方式、部署方式，也必须更新该文档中的对应章节，而不是只改代码。
5. 新增进度记录必须使用“倒序写法”：最新更新放在对应章节最上方，旧记录顺延下移，禁止把新内容追加到文档底部。
6. 所有新增记录的时间必须精确到时分秒，统一使用 `YYYY-MM-DD HH:mm:ss`（默认 `Asia/Shanghai` 时区）。

## 文档更新要求

更新 [docs/系统架构与进度.md](/Users/sam/Desktop/typeink-codex/docs/系统架构与进度.md) 时，至少维护这些内容：

- 当前阶段目标
- 系统架构思考
- 最近一次完成了什么
- 当前阻塞点或风险
- 接下来 1 到 3 个最重要动作
- 时间型记录（如“最新进度”“进行中的工作”“评估记录”）统一按倒序维护：最新日期始终在最前面。
- 时间标题必须写到秒（例如：`2026-04-06 14:23:58`），禁止只写日期。

## 开发原则

1. 先做可运行的最小闭环，再接真实 ASR、真实 LLM、真实移动端。
2. 所有实时链路优先保证“不断流、不阻塞、可观测”，再谈模型效果。
3. 遇到需要安装依赖、调整环境、删除文件、改动范围很大时，先说明影响，再继续。
4. 新增给用户看的文档尽量使用中文命名；但像 `AGENTS.md`、Python 包目录、模块文件这类受工具和框架约束的命名，按英文规范处理。

## GitHub 发包规则

1. 每次准备提交并推送 GitHub 前，至少重新执行一次 `./tools/build_apk.sh <版本号>`，生成最新重命名 APK。
2. 当前唯一 APK 构建结果目录是 `outputs/`，不要再创建或依赖 `发布包/` 这类平行目录。
3. 如果 GitHub 普通仓库文件大小限制导致 APK 不能直接进仓库历史，仍然要完成体验包上传：
   - 代码正常提交和推送；
   - 从 `outputs/` 里选最新 APK 上传到 GitHub Release 附件，不能漏掉。
4. README 只负责告诉用户“去哪里拿体验包”，不负责约束 AI 助理的发包动作；发包执行规则统一写在 `AGENTS.md`。
