# Pages as Code — Plugin Development

## What this is

WordPress plugin that provides file-backed Gutenberg pages. `.html` files with YAML front matter + block markup are pushed to WordPress via `wp pac push`.

## Architecture

```
pages-as-code.php           Bootstrap, activation hook, WP-CLI registration
includes/
  class-pac-file.php         File parsing: front matter splitting, YAML parser, slug resolution, asset resolution
  class-pac-pusher.php       Create/update pages, hash comparison, meta writes (incl. asset meta)
  class-pac-cli.php          WP-CLI commands: push, pull, validate; argument handling, output formatting
  class-pac-puller.php       Extract pages from WordPress: query, serialize, file write, revision tracking
  class-pac-serializer.php   YAML front matter + block body serialization (shared by pull, future tools)
  class-pac-validator.php    Block markup validation: parse_blocks() tree walker, per-block rules, JSON reports
  class-pac-assets.php       Frontend + editor CSS/JS enqueue for managed pages
.claude/skills/pages-as-code/   Skill shipped to users (copied to pages dir on activation)
assets/pages-CLAUDE.md       Agent instructions shipped to users (copied to pages dir on activation)
tools/generate-readme.php    Generates readme.txt from README.md + readme.meta.json
```

## Key files

- **README.md** is the canonical documentation. Edit this, not readme.txt.
- **readme.txt** is auto-generated by `tools/generate-readme.php`. Do not edit directly.
- **readme.meta.json** holds WordPress-specific metadata (stable tag, contributors, etc.). Update this when bumping versions.
- **assets/pages-CLAUDE.md** is copied to `wp-content/pages/CLAUDE.md` on activation. This is the user-facing agent instructions file.
- **CLAUDE.md** (this file) is for plugin development only. It is NOT copied to the pages directory.

## Version bumping

When releasing a new version, update all three locations:
1. `pages-as-code.php` — plugin header `Version:` and `PAC_VERSION` constant
2. `readme.meta.json` — `stableTag` field
3. `CHANGELOG.md` — add new version entry

The readme.txt is regenerated automatically by the pre-commit hook.

## Conventions

- WordPress Coding Standards (WPCS): tabs, Yoda conditions, snake_case functions
- All post meta keys prefixed `_pac_` (underscore = hidden from custom fields UI)
- Capability requirement: `edit_pages`
- Only `page` post type — no posts, no CPTs
- Zero external dependencies — bundled minimal YAML parser
- No admin UI, no settings page, no dashboard widgets

## Testing workflow

```bash
# Lint PHP
php -l pages-as-code.php && php -l includes/*.php

# Test push cycle
wp pac push test-page.html --user=1
wp pac push test-page.html --user=1   # Should report "unchanged"

# Validate block markup
wp pac validate test-page.html --user=1   # JSON report, exit 0 if ok
wp pac validate test-page.html --user=1 --strict  # Warnings also cause exit 1

# Pull a page from WordPress
wp pac pull about --user=1                # Writes pages/about.html
wp pac pull about --dir=drafts/ --user=1  # Writes pages/drafts/about.html
wp pac pull about --revision-suffix --user=1  # Writes pages/about.r123.html
wp pac pull about --force --user=1        # Overwrites existing file

# Validate skill
python3 ~/.claude/skills/skill-creator/scripts/quick_validate.py .claude/skills/pages-as-code
```

## Readme pipeline

README.md is the source of truth. The pre-commit hook runs `tools/generate-readme.php` to regenerate readme.txt from README.md + readme.meta.json, then stages the result.

To regenerate manually: `php tools/generate-readme.php`

## What gets copied to wp-content/pages/ on activation

- `assets/pages-CLAUDE.md` -> `wp-content/pages/CLAUDE.md`
- `.claude/skills/` tree -> `wp-content/pages/.claude/skills/`
- `.gitkeep` -> `wp-content/pages/.gitkeep`
