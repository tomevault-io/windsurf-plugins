## Workflow

### 1. Plan

作業前に、短い方針を共有する。
必要なら以下を明確にする。

* 何を直すか
* どのファイルを触るか
* どう検証するか

### 2. Execute

* 変更は必要最小限にする
* 無関係なファイルは触らない
* 既存規約や命名に合わせる
* 仕様変更が必要なら、関連文書の更新も検討する

### 3. Verify

可能な限り再現コマンド・確認手順を実行する。

例:

* `npm run build`
* `npm run test`
* `npm run lint`
* 手動確認

### 4. Report

完了時は最低限以下を報告する。

* 何を変更したか
* なぜ変更したか
* どう検証したか
* 残っている注意点

## Task Management

非自明タスクでは、以下を使うこと。

* `.codex/tasks/todo.md`
* `.codex/tasks/lessons.md`
* `実装履歴.md`

### ルール

* 計画は `.codex/tasks/todo.md` にチェックリスト化する
* 進捗と検証ログも同ファイルに追記する
* ユーザー修正や方針変更が入ったら、再発防止策を `.codex/tasks/lessons.md` に追記する
* 完了した非自明タスクは、人向け要約を `実装履歴.md` に追記する

---

## Editing Rules

* 影響範囲は最小化する
* 無関係ファイルを変更しない
* 既存規約 / 命名に合わせる
* 必要以上のリファクタをしない
* 一時しのぎではなく、根本原因を優先する

---

## Core Principles

### シンプルさを第一に

可能な限り単純な実装を選ぶ。
複雑な抽象化や早すぎる拡張は避ける。

### 怠惰を許さない

表面的な対症療法ではなく、原因を特定して直す。

### 影響を最小限にする

必要な変更だけを行い、既存挙動の破壊を避ける。

## Verification

* 可能な限り再現コマンドを実行する
* 成否ログを要約して報告する
* 高リスク変更では、差分意図を明記する

### 高リスク変更の例

* Save/Load
* 数式
* Prestige条件 / リセット処理
* Automation条件
* 描画ループやパフォーマンス周り

## Communication

## Definition of Done

以下を満たして初めて完了とする。

* 要求された変更が入っている
* 影響箇所を検証している
* 検証結果を報告している
* 無関係ファイルを極力触っていない
* 必要なら `.codex/tasks/todo.md` を更新している
* 必要なら `.codex/tasks/lessons.md` を更新している
