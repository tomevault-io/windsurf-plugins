# Copilot Instructions

This repository contains shared, reusable GitHub Actions workflows and composite actions for **Team BAKS** at NAV, targeting apps deployed on [Nais](https://doc.nais.io/).

## Architecture

There are two types of building blocks:

- **Reusable workflows** (`.github/workflows/*.yaml`) — triggered exclusively via `workflow_call`. Consumer repos call these by reference.
- **Composite actions** (`.github/actions/*/action.yaml`) — lower-level building blocks composed into the reusable workflows above.

Consumer repos reference both at `@main`:
```yaml
uses: navikt/familie-baks-gha-workflows/.github/actions/setup-java-maven@main # ratchet:exclude
uses: navikt/familie-baks-gha-workflows/.github/workflows/deploy.yaml@main
```

There are no local build, test, or lint commands — this is purely YAML configuration.

## Action Pinning with Ratchet

All third-party actions must be pinned to a full commit SHA with a version comment. The tool [ratchet](https://github.com/sethvargo/ratchet) manages these pins:

```yaml
# ✅ Third-party — always pin to SHA
uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # ratchet:actions/checkout@v6

# ✅ Internal self-references — excluded from ratchet pinning
uses: navikt/familie-baks-gha-workflows/.github/actions/deploy@main # ratchet:exclude

# ❌ Never use unpinned tags for third-party actions
uses: actions/checkout@v4
```

Dependabot (weekly, Mondays) opens grouped PRs to bump action SHAs.

## Key Conventions

**All workflows are `workflow_call` only** — no `push`, `schedule`, or other triggers. Every workflow file is a library entry point for consumer repos.

**Harden Runner is included in every production workflow** (not PR/ktlint jobs):
```yaml
- name: Harden Runner
  uses: step-security/harden-runner@<SHA> # ratchet:step-security/harden-runner@v2
  with:
    egress-policy: audit
```

**Timezone is always set explicitly**:
```yaml
env:
  TZ: 'Europe/Oslo'
```

**Permissions are always minimal and explicit per job**, with comments explaining why each permission is needed:
```yaml
permissions:
  contents: read   # fetch repo
  id-token: write  # nais/deploy or nais/docker-build-push
```

**Image registry** for all team apps:
```
europe-north1-docker.pkg.dev/nais-management-233d/teamfamilie/<repo-name>
```

**Nais clusters**: `dev-fss`, `dev-gcp`, `prod-fss`, `prod-gcp`

## Workflow Catalogue

| Workflow | Purpose |
|---|---|
| `build-maven-app.yaml` | Build + optionally push Maven/Kotlin Docker image |
| `build-yarn-app.yaml` | Build + optionally push Yarn/Node Docker image |
| `pull-request-maven.yaml` | Run Maven verify on PRs, optionally upload coverage |
| `pull-request-yarn.yaml` | Run Yarn build/test on PRs |
| `deploy.yaml` | Deploy a specific image to a Nais cluster |
| `deploy-with-tag.yaml` | Manual deploy by image tag (constructs full path from repo name) |
| `scan-vulnerabilities-maven.yaml` | CodeQL + Trivy scan for Maven apps |
| `scan-vulnerabilities-yarn.yaml` | CodeQL + Trivy scan for Yarn apps |
| `sonar.yaml` | Sonar analysis, expects coverage artifacts from prior jobs |
| `ktlint.yaml` | Kotlin linting via Maven `antrun:run@ktlint` |
| `notify-slack.yaml` | Post failure notification to Slack via webhook |
| `digestabot.yaml` | Auto-bump Chainguard Docker image SHAs |

## Maven specifics

- Uses `maven-settings.xml` (generated by `setup-java-maven`) to authenticate against `maven.pkg.github.com/navikt/familie-felles`
- Maven command defaults to `install`; PR workflow uses `verify`
- Flaky tests are retried automatically: `-Dsurefire.rerunFailingTestsCount=2`
- Coverage uses the `coverage` Maven profile (`-Pcoverage`)
- `byosbom` input supports bringing a custom SBOM for nested dependency scanning

## Trivy and Security Scanning

The `trivy` composite action outputs SARIF to GitHub Security tab (requires `security-events: write`). It uses multiple DB mirrors as a workaround for upstream availability issues:
```yaml
TRIVY_DB_REPOSITORY: public.ecr.aws/aquasecurity/trivy-db,aquasec/trivy-db,ghcr.io/aquasecurity/trivy-db
```

## Boundaries

**Always:**
- Pin all third-party action references to full SHA with `# ratchet:ActionName@vX` comment
- Mark internal self-references with `# ratchet:exclude`
- Set explicit `permissions` per job with inline comments
- Use `step-security/harden-runner` with `egress-policy: audit` in deployed workflows

**Ask first:**
- Adding new secrets or environment variables
- Changing deploy ordering (dev → prod)
- Adding new reusable workflows or composite actions

**Never:**
- `permissions: write-all`
- Unpinned third-party action versions (`@v4`)
- Log secrets in workflow output
- Use `pull_request_target` with checkout of PR branch
