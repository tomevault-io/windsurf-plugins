# Project Constitution

> このドキュメントはプロジェクトの原則と要件定義を記録したものです。

## プロジェクト概要

### ビジョン

dress-cli は、ターミナル上で Markdown を美しく描画する CLI ツール。
glow のように Markdown をレンダリングし、less コマンドのようなページング表示と vim キーバインドによる移動・検索をサポートする。

### ターゲットユーザー

- 個人プロジェクト（自分用）
- ターミナルを日常的に使う開発者

---

## 核となる原則

### 設計原則

1. **パフォーマンス優先**: 描画速度やメモリ効率を重視する。開発速度より実行時性能を優先
2. **シンプルさ優先**: 機能を絞ってシンプルに保つ。柔軟性より分かりやすさ
3. **外部依存最小限**: 依存ライブラリを極力減らす。必要なものだけ入れる

### トレードオフの判断基準

| 優先する | より | 理由 |
|---------|------|------|
| パフォーマンス | 開発速度 | CLI ツールは体感速度が命 |
| シンプルさ | 柔軟性 | 個人ツールなので必要十分でいい |
| 安定性 | 最新技術 | 保守的なバージョン管理方針 |

### やってはいけないこと

- **不要な外部依存の追加**: 標準ライブラリや既存の依存で実現できることに新しいクレートを入れない

## 技術スタック

### 採用技術

| カテゴリ | 技術 | 選定理由 |
|---------|------|---------|
| 言語 | Rust | パフォーマンスとメモリ安全性。CLI ツールに最適 |
| TUI フレームワーク | ratatui | Rust TUI の定番。豊富なウィジェット |
| Markdown パーサー | comrak | GitHub Flavored Markdown 対応 |
| エラーハンドリング | anyhow + thiserror | 定番の組み合わせ。anyhow でアプリ層、thiserror でライブラリ層 |

### アーキテクチャ

- シンプルに保つ。特定のアーキテクチャパターンに縛られず、必要に応じて構造化する

---

## コーディング規約

### フォーマット・リンター

- **rustfmt**: 標準設定
- **clippy**: 標準設定

### コメント

- コメントは日本語で書く
- "what" ではなく "why" を書く。自明なコードにはコメント不要

### エラーハンドリング

- アプリケーション層: `anyhow::Result` で簡潔に
- ライブラリ層（公開 API）: `thiserror` で型安全なエラー型を定義
