---

## 2. 项目目标

为公司的商城AI客服系统，构建一个高性能、高可扩展的**MCP (Machine Control Program) 服务**。

本项目聚焦于 MCP 服务的开发。该服务是 AI Agent 与后端业务系统（如 Java 商城 API）之间的核心桥梁，其主要职责是：将复杂的内部业务能力，封装成清晰、标准的工具接口，供 AI Agent 动态调用。

## 3. 我的技术背景

-   **主要技能**: Go后端开发、Linux运维。
-   **熟悉框架**: 熟练使用 Go 的 Gin 框架。
-   **兴趣方向**: 学习并实践AI相关知识，特别是MCP应用和AI编排服务。

## 4. 系统架构

在整个AI客服体系中，各组件围绕 MCP 服务协同工作。

### 4.1. MCP 服务 (本项目)
-   **角色**: **业务执行核心** 与 **动态工具提供者**。
-   **作用**: 封装所有（Java API）的业务逻辑，并通过标准接口向客户端提供原子化的业务能力。
-   **SDK**: 使用`modelcontextprotocol/go-sdk`项目进行MCP项目开发

### 4.2. AI 编排服务 (客户端)
-   **角色**: MCP 服务的主要客户端。
-   **作用**: 负责处理用户对话、进行意图理解。当需要执行具体业务操作时，它会查询并调用 MCP 服务提供的工具。

### 4.3. 其他组件
-   **客服平台 (Chatwoot)**: 用户交互的前端界面。
-   **底层技术**: 包括 LLM、向量数据库等，由 AI 编排服务负责调用。

## 5. MCP 服务核心交互模式

MCP 服务通过定义一套清晰的交互模式，与客户端（AI 编排服务）进行解耦和协作。

### 5.1. 动态能力发现 (Dynamic Capability Discovery)
-   **目的**: 该接口返回一个 JSON 格式的“能力说明书”，详细描述了 MCP 所能执行的所有业务工具，包括每个工具的名称、功能说明、以及调用时所需的参数。
-   **机制**: 客户端在启动时或定期调用此接口，即可自动学习和更新其可用的工具集，实现了能力的“即插即用”。MCP 服务每次新增或修改工具，客户端无需任何代码改动即可适配。

### 5.2. 业务工具调用 (Business Tool Invocation)
-   **接口**: MCP 根据 “能力说明书” 中定义的工具，提供对应的业务 API 接口（例如 `POST /api/v1/orders/query`）。
-   **机制**: 客户端在理解用户意图后，根据从 “能力说明书” 获取的规范，构造一个标准的 HTTP 请求来调用相应的业务接口，并在请求体中传递必要的参数。

### 5.3. 流式响应机制 (Streamable HTTP)
-   **目的**: 针对查询订单详情等可能耗时较长、需要调用多个下游服务的复杂业务，MCP 采用流式响应来优化客户端的等待体验。
-   **机制**: 在收到客户端的业务调用请求后，MCP 不会等待所有数据都准备就绪再返回，而是：
    1.  立即与客户端建立 HTTP 长连接。
    2.  一旦获取到部分结果（如订单基本信息），就立即将其作为第一个数据块（一个独立的 JSON 对象）发送给客户端。
    3.  随后，继续执行后续任务（如查询商品列表、调用物流 API），并将每一个后续结果作为新的数据块，依次、实时地推送给客户端。
    4.  所有数据块发送完毕后，关闭连接。
-   **优势**: 这种方式将“一次性”的重度等待，分解为“多次”的轻量接收，使系统的表观响应速度得到极大提升。

### 5.4. 授权与安全 (Authorization & Security)

为实现用户级别的权限控制（如查询个人订单），项目采用基于 JWT 的零信任安全模型。

### 简化流程：

1.  **获取会话令牌**: 用户登录后，前端从**商城后端**获取一个临时的、包含 `user_id` 的 `ChatSessionToken` (JWT)。
2.  **注入会话**: 前端将 `ChatSessionToken` 注入到聊天工具 (Chatwoot) 的会话属性中。
3.  **传递令牌**: **AI Agent** 在调用 MCP 时，从会话属性中提取 `ChatSessionToken`，并放入请求头的 `Authorization` 字段。
4.  **MCP 验证**: **MCP 服务**的中间件使用**商城公钥**对 `ChatSessionToken` 进行验签，成功后解析出 `user_id` 并存入请求上下文。
5.  **最终授权**: MCP 调用**商城接口**时，必须同时传递业务参数和 `user_id`，由商城完成最终的数据权限校验（例如 `WHERE order_id = ? AND user_id = ?`）。


## 6. Go 编码规范 (必须遵守)

### 6.1. 语法与性能

*   **字符串拼接**: 对于多个字符串的拼接，必须使用 `strings.Builder` 来提升性能。
*   **预分配内存**: 使用 `make` 创建 slice 或 map 时，应根据预估大小设置初始容量，以减少内存重新分配。
*   **大 Slice 切片**: 从大 slice 中切取一小部分后，建议使用 `copy` 创建新 slice，以帮助垃圾回收器释放不再使用的内存。
*   **JSON 标签**: 熟练使用 `json:"-"` 来忽略字段，以及 `json:"fieldName,omitempty"` 来忽略空值字段。
*   **go协程**: 尽量使用go协程提高并发, 分析是否有必要使用到`chan`。
*   **事务**: 事务内非必要不做查询操作。
*   **命名**: 变量,常量,文件 等命名务必要简洁且规范, 同时避免使用单个字母的命名。

### 6.2. 并发安全

*   **协程管理**: 推荐使用 `errgroup` 来管理一组协程的生命周期和错误。
*   **Panic 处理**: 任何 `go` 启动的协程内部，都必须使用 `defer` 和 `recover` 来捕获潜在的 `panic`，防止主程序崩溃。
*   **循环闭包**: 在 `for` 循环中启动协程，必须将循环变量作为参数传入，以避免闭包陷阱。
*   **单次执行**: 使用 `sync.Once.Do` 确保配置加载等初始化操作只执行一次。
*   **并发读写**: 读多写少的场景优先使用 `sync.RWMutex`。
*   **数据竞争**: 使用协程时, 避免数据竞争和闭包陷阱,确保内存安全。

### 6.3. 健壮性与设计模式

*   **依赖注入 (DI)**: 优先通过函数参数传递依赖（如数据库连接），严格避免在函数内部调用全局变量。
*   **上下文 (Context)**: 所有可能阻塞或涉及 I/O 的函数，其第一个参数都应是 `context.Context`，用于控制超时和取消。
*   **错误处理**:
    *   必须对所有可能返回 `error` 的函数进行检查。
    *   查询数据库后，必须显式处理 `sql.ErrNoRows` 的情况。
*   **Map 安全**: 读取 map 时，必须使用 `value, ok := m["key"]` 的双返回值形式进行安全判断。
*   **空指针防御**: 对配置文件、反序列化等外部数据源保持警惕，做好 `nil` 检查。

### 6.4. 禁止事项

*   **禁止 `fmt`**: 在业务逻辑代码中，使用项目配置的结构化日志库，禁止使用 `fmt` 包打印日志。
*   **禁止滥用 `init`**: 尽量将初始化逻辑放在 `main` 函数或构造函数中。
*   **禁止可变全局变量**: 避免使用任何可能在运行时被修改的全局变量。
*   **注意 Map 无序**: 不要编写任何依赖 `for range` 遍历 `map` 顺序的代码。

## 7. 其他
