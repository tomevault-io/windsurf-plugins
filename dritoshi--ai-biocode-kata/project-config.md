# CLAUDE.md — 書籍執筆プロジェクト

## プロジェクト概要
- 書籍タイトル:『AIエージェントと学ぶ バイオインフォマティクスプログラミングの作法』
- サブタイトル: 配列解析から機械学習まで、環境構築・テスト・設計・公開のベストプラクティス
- 対象読者: プログラミング初心者の実験系生命科学者
- 言語: 日本語

## ファイル構成
- `chapters/` — 各章のMarkdownファイル
- `README.md` — 全体構成・目次・節構成の一覧
- `chapters/hajimeni.md` — はじめに（番号なし）
- `figures/` — 図表
- `references/` — BibTeXファイル（章ごとの参考文献管理）
- `scripts/` — 書籍内のコードサンプル（動作テスト対象）
- `tests/` — サンプルコードのテスト
- `build/` — PDF/EPUB生成スクリプト

### ファイル命名規則
ファイル番号はREADME.mdの目次の章番号と一致させる。「はじめに」は番号なし。

| 対象 | 命名パターン | 例 |
|------|------------|-----|
| はじめに 本文 | `chapters/hajimeni.md` | — |
| はじめに 参考文献 | `references/hajimeni.bib` | — |
| §N 本文 | `chapters/NN_name.md` | `chapters/00_ai_agent.md`（§0） |
| §N 参考文献 | `references/chNN.bib` | `references/ch00.bib`（§0） |
| §N コードサンプル | `scripts/chNN/` | `scripts/ch01/`（§1） |
| §N テスト | `tests/chNN/` | `tests/ch01/`（§1） |

## 執筆規約

### 文体
- 「である」調（敬体ではなく常体）
- ただし読者への語りかけが自然な箇所では柔らかい表現も可
- 専門用語は初出時に日本語（英語）の形式で表記: 例「テスト駆動開発（Test-Driven Development; TDD）」
- 二度目以降は略称可: 「TDD」

### 章の導入構造（Vibe coding動機付け）
- 本書はVibe coding（エージェントにコードを生成させ、人間がレビュー・判断する協働スタイル）を前提とした構成である
- §0（AIエージェントとの協働）を除く各章の冒頭は、以下の3要素で構成する:
  1. **前章との接続**: 前章で学んだことを1文で振り返り、本章への橋渡しをする。前章への相互参照リンクを付ける
  2. **エージェント協働の動機付け**: この章の知識が、AIエージェントとの協働でなぜ必要かを具体的に示す。以下の2層で構成する:
     - **価値の提示**: この知識によって読者のエージェント協働実践がどう拡がるか（「この知識があれば〜ができるようになる」「この知識がなければ〜に限定される」）を示す
     - **役割分担**: 「エージェントが〜を生成/実行するが、人間が〜を判断/レビューする必要がある」というフレームで、具体的な判断ポイントを示す
  3. **章の概要**: 本章で扱う内容のスコープを示す
- 「Vibe coding」という用語自体は「はじめに」で定義済み。各章では用語を繰り返さず、概念（エージェントとの協働、コードレビュー、判断力）として表現する

### Markdown記法
- `**太字**` の内部にカッコ（全角 `（）`・半角 `()`）やカギ括弧（`「」`）を含めない。これらは太字の外に置く
  - ✅ `**tidy data**（整然データ）`
  - ❌ `**tidy data（整然データ）**`
  - ✅ `**プルリクエスト**「PR」`
  - ❌ `**プルリクエスト「PR」**`
  - 理由: 一部のMarkdownレンダラでカッコやカギ括弧を含む太字が正しく描画されない
- 数式表現は LaTeX インライン数式（`$...$`）で記述する
  - GitHub（MathJax）と pandoc（EPUB/PDF）の両方で正しくレンダリングされる
  - 太字やプレーンテキストではなく、常に `$...$` で囲む
  - コードブロック内のコメントではそのまま書く（`$` で囲まない）
  - 対象:
    - O記法（計算量）: `$O(n)$`, `$O(n \log n)$`, `$O(n^2)$`
    - 数学関数・演算子: `$\log n$`, `$n^2$`, `$2^{31}$`, `$10^{-3}$`
    - 変数・パラメータが数式的文脈で使われる場合: `$n$`, `$m$`, `$k$`, `$p$`
    - 不等式・大小関係: `$O(1) < O(n) < O(n^2)$`
    - 近似値: `$\approx 17$`, `$1.7 \times 10^6$`
  - 以下は `$...$` で囲まない:
    - コードブロック内のコメント
    - インラインコード（`` ` `` で囲まれた変数名やコード片）
    - 単なる数値（「10万件」「1秒」など日本語の文脈で使う数字）
  - `$...$` の直前・直後に全角カッコ `（）` を置かない（GitHub MathJax が `$` を数式区切りとして認識できなくなる）
    - ✅ `全走査($O(n)$)が必要` — 半角カッコを使う
    - ❌ `全走査（$O(n)$）が必要` — 全角カッコで数式が壊れる
    - LaTeX 内部の `{,}` も同様に GitHub 非対応: `$100{,}000$` → `$n$ = 100,000` のように数値を外に出す

### エージェントへの指示例
- 各章の各節末尾に `#### エージェントへの指示例` サブセクションを追加する
- 構造:
  1. 前置き文 — その節の内容をAI指示の文脈でなぜ重要か説明
  2. 指示例 — ブロック引用 (`>`) + カギ括弧で直接引用形式。読者がコピーして使えるレベルの具体的な指示文を2〜3個以上
  3. 必要に応じてバイオインフォ特有の罠や注意点を補足
- §0（AIエージェントとの協働）は対象外（章全体がエージェント指示の説明であるため）

### 演習問題
- 各章の末尾に `## 演習問題` セクションを設ける
- 配置: `## まとめ` の直後（`---` の後）、`## さらに学びたい読者へ` の直前
- 導入文: 「本章の内容を、エージェントとの協働を通じて実践する課題である。」
- 1章あたり 3〜5問
- 各問題の見出し形式: `### 演習 X-Y: タイトル **[類型]**`（X=章番号, Y=問題番号）
- 類型ラベル（5種）:
  - **[レビュー]** — 提示コードの問題点を指摘する
  - **[指示設計]** — エージェントへの適切な指示文を書く
  - **[設計判断]** — 技術選択やアーキテクチャの意思決定を行う
  - **[実践]** — 実際にエージェントに指示を出し、結果を評価する
  - **[概念]** — 用語や概念の理解を確認する
- 各問題の末尾にヒントを付与: `（ヒント）テキスト。`
- 模範解答は本文に含めない。`private/exercises/chNN.md` に別途配置する
- レビュー問題で提示するコードは問題文中にインラインのコードブロックとして配置する（10〜20行程度）
- §0（AIエージェントとの協働）も演習の対象とする

### コラム体系
- 🧬 — バイオインフォマティクス固有のコラム
- 🤖 — 機械学習応用のコラム
- これらのコラムは本文から独立しており、読み飛ばしても支障がないように書く

### ツール記述
- Claude Code CLI と Codex CLI の両方に対応する
- 本文では概念（「読み取り専用モード」「承認ありモード」等）で記述する
- ツール固有の操作は対照表で示す
- 特定ツールの名前をベタ書きしない（対照表の中は除く）

### コードサンプル
- Python 3.10+ を前提とする
- 型ヒントを必須とする
- サンプルコードは `scripts/` に実行可能な形でも配置し、`tests/` でテストする
- コードブロック内のコメントは日本語
- 本文中のコード例・コマンド例には、初心者が理解できるだけの説明を必ず付ける
  - コードブロックの前に「何の処理で、どのパラメータを変えているか」の文脈説明を置く
  - コマンドの主要なオプション・フラグの意味を説明する
  - 初出の関数・メソッドは「何をする関数か」を1文で説明する

### 禁止事項
- FASTAパーサの自作（Bio.SeqIOを使う）
- print()によるログ出力（loggingモジュールを使う）
- ハードコーディングされたファイルパス
- テストのないコードサンプルの掲載
- フォーマット・リンク・引用の挿入など体裁に関する指示の際に、本文の文章内容を変更すること
  - 体裁の修正のみを行い、文章の加筆・修正・削除は行わない
  - 文章内容の変更が必要な場合は、必ずユーザーに許可を求めてから行う

### 引用・参考文献
- 本文中の引用は番号方式で記載し、ハイパーリンクを付与する: 例 `[1](https://doi.org/xxx)`
- 各章の末尾に「参考文献」セクションを設け、その章で引用した文献を番号順にリストする
- 引用対象は論文・書籍・Webドキュメント（公式ドキュメント等）のすべてを含む
- 文献情報は `references/` ディレクトリ内の BibTeX ファイル（`.bib`）で一元管理する
- 章ごとに `references/chNN.bib` を作成する（NNは章番号。例: `references/ch00.bib` = §0）
- 参考文献リストの各エントリは以下の形式で記載する:
  - 論文: `[番号] 著者名. "タイトル". *雑誌名*, 巻(号), ページ, 年. DOIリンク`
  - 書籍: `[番号] 著者名. *書名*. 出版社, 年. ISBNまたはリンク`
  - Web: `[番号] 著者名またはサイト名. "タイトル". URL (参照日: YYYY-MM-DD)`
- 執筆時に主張・事実・ツールの根拠となる情報源がある場合は必ず引用を付ける
- AIが生成した文章であっても、引用可能な情報源が存在する場合は引用を省略しない

### 相互参照
- 章間・節間の参照には必ず相対リンクのハイパーリンクを付与する
- 章への参照: `[§X タイトル](./XX_filename.md)` — 例 `[§0 AIエージェントにコードを書かせる](./00_ai_agent.md)`
- 節への参照: `[§X-Y タイトル](./XX_filename.md#見出しのアンカー)` — 例 `[§0-2 ワークフロー](./00_ai_agent.md#0-2-plan--execute--review-ワークフロー)`
- リンクは必ず相対パスで記述する（絶対パスやURLは使用しない）
- README.md の目次の節番号を真実の源とする
- 章や節に言及するたびにリンクを付ける（初出時のみではなく常に付与する）

## Git・GitHub運用
- CLI使用時は現在のブランチにそのままコミット・プッシュする。勝手にブランチを作成しない
- ブランチの作成・切り替えはユーザーが指示した場合のみ行う
- PRの作成・マージはユーザーが行う。Claudeは行わない
- 作業完了時にはPR用のタイトルと本文案をMarkdownのコードブロックで出力する（ユーザーがコピペできるように）
