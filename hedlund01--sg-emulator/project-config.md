# GEMINI.md: sg-emulator Project

This document provides a comprehensive overview of the `sg-emulator` project, its architecture, and development conventions to be used as a technical guide for an AI assistant.

## 1. Project Overview

`sg-emulator` is a **ScaleGraph Emulator** written in **Go**. Its primary purpose is to simulate a distributed ledger/blockchain system. The project demonstrates a clean, channel-based, concurrent architecture that allows multiple transport protocols to interact with a shared application state simultaneously.

### Core Technologies

-   **Go**: The primary programming language.
-   **Protocol Buffers & ConnectRPC**: For schema-driven, type-safe communication in the gRPC transport.
-   **Chi**: A lightweight, idiomatic router for building the REST transport.
-   **Bubble Tea**: A framework for building the interactive Terminal User Interface (TUI).
-   **Model Context Protocol (MCP)**: A custom transport is implemented to allow Large Language Models (LLMs) to interact with the emulator's functions as tools.

### Architecture

The application follows a 3-layer design, ensuring a separation of concerns:

1.  **Business Logic (`internal/scalegraph`)**: This layer contains the core domain logic, including accounts, tokens, transactions, and the blockchain itself. It is pure business logic with no knowledge of servers or transport protocols.
2.  **Infrastructure (`internal/server`)**: This layer acts as the intermediary. It consists of:
    *   A central `Server` that runs the `scalegraph.App` in a dedicated goroutine.
    *   A `Registry` for managing `VirtualApp` instances.
    *   `VirtualApp`s which represent simulated nodes in the distributed network.
    *   Communication is handled via Go channels.
3.  **Transport (`internal/transport`)**: This is the outermost layer, responsible for exposing the application to the outside world. It includes implementations for:
    *   **REST (`/rest`)**: Standard HTTP API.
    *   **gRPC (`/connect`)**: RPC communication using ConnectRPC.
    *   **TUI (`/tui`)**: An interactive terminal interface for human users.
    *   **MCP (`/mcp`)**: A server that exposes application functionality to LLMs.

A **Certificate Authority (`internal/ca`)** is used to manage cryptographic identities (key pairs and certificates) for accounts, enabling transaction signing and verification.

## 2. Building, Running, and Testing

The project uses a `Makefile` to simplify common development tasks.

### Key Commands

-   **Build the application**:
    ```bash
    make build
    ```
    This compiles the main application (`cmd/app`) and the test client (`cmd/testclient`) into the `bin/` directory.

-   **Run Tests**:
    ```bash
    make test
    ```
    This command first generates mocks using `mockery` and then runs all tests using `go test`. The CI workflow also executes tests with the race detector (`go test -race`).

-   **Run the Application**: The application is started with the `bin/app` binary and configured via command-line flags.
    -   **Headless mode**: `./bin/app`
    -   **Interactive TUI**: `./bin/app -tui`
    -   **With 2 REST and 1 gRPC virtual app**: `./bin/app -rest 2 -grpc 1`
    -   **With LLM interaction server**: `./bin/app -mcp localhost:3000`
    -   **Combined mode (TUI + MCP)**: `./bin/app -tui -mcp localhost:3000`

-   **Install Dependencies**:
    ```bash
    make deps
    ```
    This downloads Go modules and installs necessary development tools like `mockery` and `swag`.

## 3. Development Conventions

### Code Generation

This project relies heavily on code generation to maintain consistency and reduce boilerplate.

-   **Protobuf (`proto/`)**: The API contracts for gRPC are defined in `.proto` files. The `buf.gen.yaml` file configures `buf` to generate Go client and server code into the `gen/` directory using `protoc-gen-go` and `protoc-gen-connect-go`.
-   **API Documentation (`internal/transport/rest/`)**: The REST API uses `swag` to generate OpenAPI (Swagger) documentation from code annotations. The generation is triggered by:
    ```bash
    make swagger
    ```
-   **Mocks (`internal/.../mocks/`)**: Unit tests use mocks generated by `mockery`. Mocks are defined in `.mockery.yaml` and generated for interfaces within the project.

### Code Structure

-   `cmd/app/main.go`: The application entry point. It handles flag parsing, logger setup, and initialization of the server and transports.
-   `gen/`: Contains auto-generated Go code from the `.proto` files. **Do not edit manually.**
-   `internal/`: Contains all the core application code.
    -   `ca/`: Certificate Authority logic.
    -   `crypto/`: Cryptographic primitives (hashing, signing, verification).
    -   `scalegraph/`: The core business logic layer.
    -   `server/`: The infrastructure layer (Server, VirtualApp, Registry).
    -   `transport/`: The transport layer (REST, Connect, TUI, MCP).
-   `proto/`: Protobuf definitions for API contracts.

### Logging

-   The application uses Go's structured logging library (`slog`).
-   In headless mode, logs are written to `stdout`.
-   When the `-tui` flag is active, logs are automatically redirected to `/tmp/sg-emulator.log` to prevent interference with the UI.
-   Log level and format (text/json) can be configured with the `-log-level` and `-log-format` flags.

### Continuous Integration

-   The CI workflow is defined in `.github/workflows/test.yml`.
-   It triggers on pushes and pull requests to the `main` and `master` branches.
-   The workflow runs on Ubuntu, checks out the code, sets up Go, downloads dependencies, runs tests (including with the race detector), and builds the application to ensure it compiles.
